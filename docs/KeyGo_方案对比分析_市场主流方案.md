## 一、KeyGo 方案完整画像

### 1.1 一句话定位

> **成本极低的 ESP32 开源 BLE 后装车钥匙** — 用 MOS 管并联原车钥匙 + BLE Bonding + 设备端 RSSI 测距，实现无感舒适进入。

### 1.2 架构全貌

```
┌──────────────────────────────────────────────────────────────┐
```

### 1.3 核心技术栈

| 层 | 技术选型 | 备注 |
|----|---------|------|
| **通信** | BLE 4.2 (ESP32-C3) | 广播扫描 + GATT 连接，双栈兼容 Bluedroid/NimBLE |
| **测距** | 设备端 RSSI + 手机端 RSSI 双向 | 固件主测量（连接 RSSI），App 辅助轮询 |
| **信号滤波** | 尖峰丢弃 + 1D Kalman 滤波 | q=4, r=16 可调参数 |
| **防抖** | 滞后状态机 + 连续确认计数器 | 解锁 3 次确认 / 锁车 5 次确认，滞后带 5dB |
| **安全** | BLE LE Secure Connections Bonding | 静态 PIN 123456，Display Only IO Cap |
| **身份** | 固件动态设备名 KeyGo-XXYYZZ | MAC 后 3 字节，NVS 持久化自定义名称 |
| **广播过滤** | 自定义 128-bit Service UUID | `0000FF00-0000-1000-8000-00805F9B34FB` |
| **持久化** | ESP32 Preferences (NVS NOR Flash) | 10 个可调参数 + PIN + 设备名 |
| **App 框架** | uni-app (Vue 3 + Pinia) | 跨端兼容 iOS/Android/微信小程序 |
| **App UI** | Swiper 自定义底部栏 | 4 tab 平滑滑动动画 |

## 二、市场方案全景图

市场上的汽车数字钥匙方案可分为三大派系：

### 2.1 前装 OEM 方案（出厂自带）

| 代表品牌 | 核心技术 | 典型车型 |
|---------|---------|---------|
| Tesla Phone Key | BLE 4.2 + 挑战-应答时间测量 | Model 3 / Y / S / X |
| BMW Digital Key | NFC (CCC R2) + BLE + UWB (CCC R3) | iX / 7系 / X1 |
| Mercedes-Benz | BLE + NFC | EQS / S 级 |
| 比亚迪 | BLE + NFC 卡片 | 汉 / 海豹 / 宋 Plus |
| 蔚来 | BLE + NFC + UWB | ET7 / ES8 |
| 华为问界 | BLE + NFC + 星闪 SLE | M7 / M9 |

**共同特点**：
- 深度集成到车辆 CAN 总线 / BCM 车身控制模块
- 多天线 RSSI 测距 (车内外区分)
- 符合 CCC 数字钥匙标准
- 云端账号 + 手机安全元件 (SE / eSE)
- 钥匙分享 + 多用户管理

### 2.2 后装改造方案（第三方加装）

| 品牌/产品 | 技术路线 | 价格区间 |
|----------|---------|---------|
| **红树河熊 X1000** | BLE + 太阳能电池 | ¥400-800 |
| **车豆荚** | BLE + 接线盒 | ¥300-600 |
| **领世车联** | BLE + 4G 远程 | ¥500-1200 |
| 淘宝杂牌 BLE 模块 | 简单 BLE 开关 | ¥50-200 |
| DIY 改装帖 | HC-05/ESP32 + 继电器 | ¥30-100 |

**共同特点**：
- **几乎全部采用手机端 RSSI 判断距离**（而非设备端）
- 控制输出方式以继电器或 MOS 管模拟按键为主
- 少数高端款集成太阳能充电
- 安全通常较弱或无安全设计
- 用户体验参差不齐：连接稳定性、扫描速度差别大

### 2.3 开源/社区方案

| 项目 | 平台 | 特点 |
|------|------|------|
| **ESPKeylessCar** (GitHub) | ESP32 | 动态学习 iPhone IRK 地址，主机模式扫描手机 |
| **esphome-tesla-ble** | ESPHome | 模拟特斯拉 BLE 钥匙协议 |
| **KeyGo (本项目)** | ESP32-C3 | 从机模式，设备端 RSSI，BLE Bonding |

## 四、通信技术路线对比

### 4.1 BLE（低功耗蓝牙）— 第二代数字钥匙核心

| 特性 | KeyGo BLE 方案 | 行业 BLE 方案 |
|------|---------------|--------------|
| 蓝牙版本 | 4.2 (ESP32-C3) | 5.0+ |
| 广播方式 | 从机持续广播 | 从机广播 |
| 连接策略 | 手机主动连接 | 手机主动连接 |
| GATT 服务 | 1 个自定义服务，3 个特征值 | 标准 CCC BLE 服务框架 |
| RSSI 测量 | **设备端** (ESP32 侧读取连接 RSSI) | **车端** (蓝牙芯片原生 API) |
| 双栈兼容 | ✅ Bluedroid + NimBLE | 通常单栈 |
| 平台兼容 | iOS / Android / 微信小程序 | iOS / Android |
| 广播包优化 | Service UUID + Mfr Data | 分层广播策略 |

**KeyGo 亮点 — "设备端 RSSI 测距"**：
- 绝大多数后装方案依赖手机端轮询 RSSI，手机锁屏后蓝牙扫描频率大幅下降，距离感知变迟钝
- KeyGo 的 ESP32-C3 在连接状态下可以以 500ms 固定间隔读取连接 RSSI（`esp_ble_gap_read_rssi` / `ble_gap_conn_rssi`），**不需要手机参与**
- 这使 KeyGo 在 App 切后台后仍然可以持续准确感知距离

### 4.2 UWB（超宽带）— 第三代数字钥匙

| 特性 | 说明 |
|------|------|
| 定位精度 | **厘米级** (~10cm)，远超 BLE 的米级 |
| 安全性 | 物理防中继攻击（飞行时间 ToF 防转发） |
| 硬件要求 | 需要专用 UWB 芯片（NXP SR200/NCJ29D5、Qorvo DW3300） |
| 成本 | 车端 + 手机端均需要 UWB 芯片 |
| 手机支持 | iPhone 11+、三星 S21+、部分国产旗舰 |
| KeyGo 可行性 | **短期内不可行**，ESP32-C3 无 UWB 模块，手机端 UWB API 也未开放给普通 App |

### 4.3 NFC（近场通信）— 备用/补充方案

| 特性 | 说明 |
|------|------|
| 通信距离 | 极近 (~4cm) |
| 典型用途 | 手机没电时贴在车门上解锁，或分享给洗车店临时用 |
| 硬件要求 | NFC 读卡器模块 (PN532/RC522) |
| 安全 | 硬件级安全（安全元件 SE） |
| KeyGo 集成难度 | **低**，增加一个 PN532 模块 + I2C 通信即可 |
| 优先级 | 中，建议 v5.0 加入 |

### 4.4 星闪 (SLE)

- 华为主导的新型短距通信技术，定位精度介于 BLE 和 UWB 之间
- 2025 年起问界 M7/M9 搭载
- 生态尚在早期，对后装方案暂不适用

## 五、测距方案对比

### 5.1 RSSI 信号处理方法

| 方案 | 滤波方法 | 滞后/防抖 | 精度 |
|------|---------|----------|------|
| **KeyGo** | **尖峰丢弃 + 1D Kalman** | **滞后带 5dB + 连续确认计数器** | ~1-3m |
| 特斯拉 | 多天线 RSSI 平均 + 时延测量 | 融合多种信号 | ~1m |
| CCC R3 | BLE 多天线 + UWB ToF | UWB ToF 天然防抖 | <0.1m |
| 红树河熊 | 推测为简单滑动平均 | 未知 | ~3-5m |
| ESPKeylessCar | 简单阈值比较 | 基本无 | ~2-5m |
| 淘宝 BLE 模块 | 原始 RSSI 单次比较 | 无 | >5m |

### 5.2 KeyGo RSSI 管线详细设计

```
蓝牙连接信号强度
      │
      ▼ (每 500ms)
┌─────────────┐
│ raw RSSI    │  手机与 ESP32-C3 之间的 BLE 连接 RSSI (dBm)
│             │  双栈读取: Bluedroid (esp_ble_gap_read_rssi) / NimBLE (ble_gap_conn_rssi)
└──────┬──────┘
       │
       ▼ (当前值距滤波值超 25dB → 暂丢弃，连续 2 次 → 放行)
┌─────────────┐
│  尖峰丢弃    │  防止身体遮挡、手机翻转等瞬间干扰
│  阈值 25dB  │  可配置: cfg spike=25
│  连续计数 2  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 1D Kalman   │  一维随机游走模型:  p = p + q  →  k = p/(p+r)  →  x = x + k*(z-x)
│  滤波       │  q=4 (过程噪声), r=16 (测量噪声)  均可调
│              │  平滑输出 filtered RSSI，稳态延迟约 1-2 次采样
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 滞后状态机   │  Locked→Unlock: filtered >= unlock 且连续 3 次满足
│ Locked ←→   │  Unlocked→Lock:   filtered <= (lock - hyst) 且连续 5 次满足
│ Unlocked    │  hyst=5dB 滞后带防止边界震荡
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 手机端辅助   │  连接状态下每 800ms 轮询广播包 RSSI
│ RSSI 上传    │  通过 FF01 注入到固件，作为设备端 RSSI 的补充
└─────────────┘
```

**行业对比评估**：
- 与特斯拉的专利多天线方案相比，精度差约 1-2m
- 但在所有**同价位后装方案**中，KeyGo 的 RSSI 管线是最完善的：有尖峰滤波、有 Kalman、有滞后状态机
- 关键差距在于**单天线 vs 多天线**：单天线无法区分车内/车外，可能出现在车外但信号强就误触发的情况

### 6.2 KeyGo 安全分析

**✅ 做得好的**：
1. **BLE Bonding 是业界标准** — 和特斯拉/BMW/比亚迪使用的是同一套链路层加密协议
2. **密钥持久化** — bond key 存储在 NVS Flash，重启不丢失
3. **PIN 可修改** — 出厂 123456 可改为自定义 6 位 PIN
4. **互斥绑定** — 短按物理按键删除所有 bond，同时只能有一台手机绑定
5. **物理恢复出厂** — 长按按键 5s 清除所有数据

**⚠️ 存在风险的**：
1. **中继攻击** — 攻击者用信号放大器可以在远处将手机信号桥接到车边，特斯拉被 NDSS 2023 论文证实存在此漏洞，KeyGo 同样面临
2. **PIN 空间太小** — 6 位数字只有 100 万种组合，暴力破解约几分钟。行业方案用云端限速或 SE 硬件来抵御
3. **没有安全元件** — 如果攻击者物理获取 ESP32-C3，可以读出 Flash 中的 bond key

### 6.3 安全对标 — 与特斯拉的差距

```
特斯拉 Phone Key 安全链:

手机 Tesla App ──(云端账号认证)──→ Tesla 云端
       │
       ├──(BLE LE Secure Connections)──→ 车端 BLE 模块
       │         │
       │    ┌────▼───────────────────┐
       │    │  挑战-应答 (Challenge) │  每次解锁/锁车动态生成
       │    │  + 时延测量 (<一定值)  │  防止中继攻击
       │    └────────────────────────┘
       │
       └──(TLS 1.3)──→ Tesla API ───→ 远程控制 (空调/定位)

KeyGo v4.0 安全链:

手机 KeyGo App ──(BLE LE Secure Connections + 静态 PIN 123456)──→ ESP32-C3
       │                              │
       │                         ┌────▼───────────────────┐
       └──(FF03 明文写入)────────→│  UNLOCK / LOCK / TRUNK│  依赖 BLE 链路加密
                                  └────────────────────────┘
```

**结论**：KeyGo 和特斯拉共享 BLE SC 链路层加密，但：
- 缺少挑战-应答层
- 缺少时延测量（中继攻击防护）
- 缺少云端鉴权
- 这些对于后装 DIY 方案的威胁模型（防路人、防邻居）暂时不重要

## 十二、与 CCC 行业标准的差距分析

CCC (Car Connectivity Consortium) 是全球车联联盟制定的数字钥匙标准，目前最新为 Release 3。
下面对比 KeyGo 与 CCC R3 的差距：

### 12.1 架构层级

```
CCC R3 标准架构:

┌────────────────────┐
│  Device (手机)      │
│  ┌──────────────┐  │
│  │ SE/eSE (安全) │  │  硬件级安全存储 (银行级)
│  │ ──────────── │  │
│  │ Applet (Java) │  │  标准密钥管理程序
│  └──────────────┘  │
│  ┌──────────────┐  │
│  │ DK App        │  │  标准 OEM/云服务 API
│  └──────────────┘  │
└────────┬───────────┘
         │ BLE (带外通信) + UWB (测距)
┌────────┴───────────┐
│  Vehicle (车辆)     │
│  ┌──────────────┐  │
│  │ BCM / ECU    │  │  车身控制模块
│  │ ──────────── │  │
│  │ UWB Anchor ×6│  │  车内多锚点精确定位
│  │ BLE Module   │  │  连接/认证/数据
│  │ NFC Reader   │  │  门把手/中控
│  │ SE (安全元件) │  │  车端安全存储
│  └──────────────┘  │
└─────────────────────┘

KeyGo v4.0 架构:

┌────────────────────┐
│  KeyGo App (手机)   │
│  ┌──────────────┐  │
│  │ uni-app BLE  │  │  ← 软件层 BLE API (无 SE)
│  │ Pinia Store  │  │
│  └──────────────┘  │
└────────┬───────────┘
         │ BLE SC (单通道)
┌────────┴───────────┐
│  ESP32-C3 (设备)    │
│  ┌──────────────┐  │
│  │ BLE Server   │  │
│  │ GATT 3特征值  │  │
│  │ RSSI 单天线   │  │  ← 1 个天线 (多天线需 ESP32-S3 以上)
│  │ Kalman 滤波   │  │
│  │ MOS 输出 ×4   │  │
│  └──────────────┘  │
└─────────────────────┘
```

### 12.2 CCC R3 与 KeyGo 差距清单

| CCC R3 要求 | KeyGo 状态 | 差距 |
|-------------|-----------|------|
| **BLE 5.0+** | ESP32-C3 支持 BLE 5.0 但 KeyGo 只用到 4.2 特性 | ⚠️ 可升级 |
| **标准 CCC BLE Service** | 自定义 FF00 服务 | ❌ 需重写 |
| **标准 CCC UUID** | 自定 FF01/FF02/FF03 | ❌ 需重写 |
| **标准帧格式 (DK Message)** | 自定 key=value / JSON | ❌ 需重写 |
| **UWB 测距模块** | 无 | ❌ 芯片不支持 |
| **车端 Secure Element** | 无 | ❌ 需硬件 |
| **手机端 SE/eSE** | 无 | ❌ 手机硬件限制 |
| **非对称证书认证** | 静态 PIN + SC Bonding | ⚠️ 安全级别低两级 |
| **标准化钥匙分享** | 无 | ❌ 需完整 CCC 协议栈 |
| **多天线定位** | 单天线 | ⚠️ 芯片限制 |
| **NFC 备用** | 无 | ❌ 可加 PN532 |

### 12.3 是否应该追求 CCC 兼容？

| 因素 | 分析 |
|------|------|
| **必要性** | 后装 DIY 方案不需要 CCC 认证，CCC 是前装 OEM 的游戏规则 |
| **可行性** | 完整 CCC R3 合规需要 SE 芯片 + UWB + 标准协议栈，硬件成本 ×100 |
| **投入产出比** | 对 KeyGo 项目极度不划算 |
| **建议** | **不追求 CCC 合规**，而是学习其设计思想（多天线、挑战应答、标准帧格式），在 KeyGo 能力范围内优化 |

## 十三、发展路线图建议

基于对比分析，建议 KeyGo 按以下优先级发展：

### 短期 (v4.1 ~ v4.5) — 补齐关键短板

| 优先级 | 功能 | 依据 |
|--------|------|------|
| P0 | **多手机绑定支持** | 对标红树河熊/特斯拉，家庭场景核心需求 |
| P0 | **NFC 解锁备用** | 对标 CCC R3/特斯拉，手机没电的救命方案 |
| P1 | **广播包序列号 (方案A)** | 解决 deviceId 不可靠问题，唯一识别设备 |
| P1 | **OTA 固件升级 (通过 BLE)** | 对标商用方案，无需拆机烧录 |
| P1 | **太阳能充电** | 对标红树河熊，长期免维护 |

### 中期 (v5.0) — 体验升级

| 功能 | 说明 |
|------|------|
| **钥匙分享** | 对标红树河熊 App 的分享功能，生成限时/限次使用码 |
| **挑战-应答安全层** | 在 FF03 命令之上加一层动态挑战码，降低中继攻击风险 |
| **RSSI 实时曲线** | App 新增 RSSI 历史曲线图，辅助调参和问题排查 |
| **操作历史日志** | 记录解锁/锁车时间戳，车主导航 |

### 长期 (v6.0+) — 生态扩展

| 功能 | 说明 |
|------|------|
| **M5Stack 屏幕版** | 车内中控显示，独立于手机运行 |
| **语音控制** | 集成语音助手，说"开锁"即可 |
| **4G 远程模块** | 对标 4S 店安吉星，千里之外远程控车 |
| **行车记录联动** | 触发解锁时自动启动记录仪 |
| **多车型适配数据库** | 社区贡献不同车型的 GPIO/时序参数 |
