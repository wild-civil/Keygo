## 1. 固件现状事实核查（结论依据）

### 1.1 电池电压：内部 VBAT 通道，≈30s 一次，无外部分压 / 无 MOS

来源：`code/CH582M/CH582M_BLE_Slave/Profile/battery_service.c`

- `Battery_UpdateLevel()` 读取 **CH582M 内部 VBAT 通道（channel 14，`CH_INTE_VBAT`）**。
- 注释明确：“通过内部 VBAT ADC 通道读取，不需要外部引脚”。
- 触发：`SBP_BATTERY_CHECK_EVT` 周期 `SBP_BATTERY_CHECK_PERIOD = 48000`（tmos tick，约 **30s** 一次）——所以**不是 5s**，比温度还稀疏。
- 采样瞬间：`R8_ADC_CFG = RB_ADC_POWER_ON | …` 上电 → 启动转换 → 等完成 → **读完后立刻把 `R8_ADC_CFG` / `R8_ADC_CHANNEL` 恢复原值**（ADC 采样完即关闭）。
- **没有外部电阻分压、没有 MOS 开关**——量的是芯片自身 VDD，内部通道自带，零外部器件。
- ⚠️ 现实问题：当前 LDO@3.3V 板子上，内部 VBAT 永远读到 ≈3.3V，电量百分比几乎不变（固件 TODO 自述“固定 3.3V 则读不到 4.2V”变差”）。**现在的电量显示意义有限。**

### 1.2 芯片温度：内部 TSENSE，5s 节流单次

来源：`code/CH582M/CH582M_BLE_Slave/APP/keygo_core.c:851`

- `KeyGo_ReadTemperatureC()`：`HAL_GetInterTempValue()` 触发一次内部温度传感器 ADC 采样 + `adc_to_temperature_celsius()` 按 ROM 出厂校准换算。
- `#define TSENSE_SAMPLE_INTERVAL_MS 5000`：**5s 节流缓存**，仅在组 FF02 状态包时调用一次。
- 平时根本不采样，单次转换 µs 级。

### 1.3 LED：PB4 → 1kΩ 限流 → LED → GND

来源：`code/CH582M/CH582M_BLE_Slave/APP/keygo_core.c:232`

- 电路：`PB4 → 1kΩ 限流电阻 → LED 正极 → LED 负极 → GND`。
- PB4 配 `GPIO_ModeOut_PP_5mA`，但电流被 1kΩ 限制：  
  `I = (3.3V − Vf) / 1000Ω`，红灯 Vf≈1.9V → **≈1.4mA**；蓝/绿灯 Vf≈2.8V → **≈0.5mA**。
- 固件注释自带“低功耗提示”：量产前关 LED，搜 `[LED_BEGIN]`/`[LED_END]` 标记删除即可。

## 2. 功耗量级对比（核心数值）

| 项 | 活动/导通电流 | 在 KeyGo 下的**平均电流** | 备注 |
|---|---|---|---|
| **TSENSE 内部温度** | 几 µA（仅 ~10µs 转换期） | **≈ pA ~ nA 级** | 5s 节流，可忽略 |
| **LED（PB4, 1kΩ）** | **~0.5 ~ 1.5mA**（亮时） | 常亮 1.5mA / 闪烁（1% 占空）≈14µA | 比 TSENSE 大 **约 6 个数量级** |
| **外置电池分压（无 MOS）** | 常通漏电 | **~20µA**（4.2V / 200kΩ 总阻） | **真实耗电源** |
| **外置电池分压（有 MOS）** | 仅采样 ms 内导通 | ~nA 级 | MOS 干掉常通漏电 |
| **外置数字温度传感器（I2C，如 TMP102）** | 活动 40µA / 关断 <0.5µA | ~0.2µA（5s 采样 26ms） | 极小 |

**排序（从费电到省电）**：
`LED 常亮` ≫ `无 MOS 外置分压` ≳ `外置 I2C 传感器` > `TSENSE 内部温度`

➡️ **TSENSE 内部温度是整张表里最不耗电的一项**。为“省电”关它，省不出任何可测量电量（纯安慰剂）。

## 3. 用户三问的讨论

### Q1：电池电压现在是“定时 5s 采集”还是“MOS 控制采集”？
**都不是。** 当前是**内部 VBAT 通道、~30s 定时、ADC 采样完即关**，无外部分压、无 MOS。

“MOS 控制采集”是**外部分压场景**才需要的技术：
- 只有当想测“电芯真实电压”、而芯片被 LDO 固定在 3.3V（内部通道只能看到恒定的 3.3V，看不出电芯衰减）时，才需外接分压电阻接到电芯。
- 分压电阻会**持续漏电**（如 4V/200kΩ ≈ 20µA 一直流），这时用 **MOS 在采样瞬间导通、平时关断**来省掉这个常通漏电。
- 当前板子走内部通道，这条路未启用。等后期画板改外置通道时，才需要认真设计分压 + MOS。

### Q2：温度采集、电池采集耗电吗？
对当前内部通道设计，**两者平均电流都≈0（nA 级）**，相对 BLE 射频（连接/广播 **mA 级峰值**）是可忽略的零头。

**唯一的真实耗电隐患在“外部电路”**：
- 若将来为测准电芯电压改用**外部分压电阻（无 MOS 控制）**，分压电阻常通漏电才是大头。
- **MOS 控制就是干掉这个常通漏电的唯一有效手段**——但仅在外置通道场景下成立。

### Q3：之前提的“细化实现方案”，做成『显示类』还是『彻底关掉』？
这是两种层级，省电效果天差地别：

- **A. 仅显示类（App 端隐藏）**：固件照常采样，只是 App 不显示。→ **省电 = 0**（固件仍在 ADC 转换）。意义仅“界面干净 / 用户不想看”。
- **B. 彻底关掉（固件级）**：加 DataFlash 位，固件跳过 `Battery_UpdateLevel()` / `KeyGo_ReadTemperatureC()`，真不再转换。→ 对当前内部通道仍是 ≈0；但**若将来上外部分压 + MOS 方案，关掉才能省掉常通漏电**。

结论：当前内部通道下，A、B 省电都≈0。**若只为功耗，加开关无意义。** 要加就做 **B（固件级彻底关）** 才名副其实；做 A 会误导用户“关了能省电其实没省”。

## 5. 当前决策（锁定）

- **暂不改代码。** 原因：TSENSE 本就≈0，关它省不出电量，属于安慰剂。
- “温度可关”若实施，仅定位为**可选特性 / 用户选择权 / 展示可关**，绝不当作省电开关宣传。
- 真正要管的“超级省电”杠杆排序：
  1. **BLE 射频参数**（广播间隔、连接间隔、从机延迟）——最大头。
  2. **LED**（或闪烁指示，必要时按 `[LED_BEGIN]/[LED_END]` 全删）。
  3. **外置分压 MOS 控制**（画板后外置电压通道必做）。
  4. **CPU 深度睡眠 / 外设时钟门控**。
  5. 温度 / 内部电压这类 ADC 采样——**最末位，可忽略**。

## 6. 未来方向讨论（设想 / 蓝图）

### 6.1 外置电压采集（画板后）
- 改用**外部分压电阻**接到电芯，配合 **MOS 管**在采样瞬间导通、平时关断。
- 目的：在 LDO@3.3V 下也能读到电芯真实电压，电量百分比才有意义。
- 关键设计：
  - 分压总阻尽量大（如 ≥200kΩ）降低漏电；MOS 选低漏电小信号管。
  - 采样流程：开 MOS → 延时稳定 → ADC 采样 → 关 MOS。
  - 复用现有 `Battery_UpdateLevel()` 节流周期（30s 足够，电量变化慢）。
- 这是“超级省电”里**唯一和电压采集相关的真实省电点**（消灭常通漏电）。

### 6.2 温度采集的“可选特性”抽象（可移植性）
- 纠正一个前提：**nRF52832 / nRF52840 都内置 TEMP 温度传感器外设**（−40~85°C，校准后 ±0.5°C）。所以“换 nRF52 就没内部温度”不成立——真移植到 nRF52，直接不实现 TSENSE 那段即可，**不需要运行时开关**。
- 合理的抽象是两层：
  - **编译期**：`#ifdef KEYGO_HAS_TSENSE`——无内部温度的 MCU 直接编译掉温度代码（最省、最干净）。
  - **运行期**：CH582M 上保留 `tempEn` 开关（复用 CFG / DataFlash），满足“用户可关 / 展示可关”需求。
- 对外置数字温度传感器的支持（若某 MCU 完全无内部温度且要求测温）：I2C（TMP102 类）平均电流 ~0.2µA，亦可经 `tempEn` 门控。

### 6.3 超低功耗 / 超级省电模式蓝图
- 目标：长期待机（电瓶车可能停 weeks~months）下的续航最大化。
- 杠杆（按影响从大到小）：
  1. **射频**：非连接期用长广播间隔 + 深睡；连接期协商大连接间隔 + 从机延迟。
  2. **LED**：默认关或仅在事件时短闪。
  3. **外置分压 MOS**：§6.1。
  4. **CPU**：CH582M 进 idle/hold 低功耗模式，靠 RTC / BLE 事件唤醒。
  5. ADC 类采样（温度/电压）：节流到 5s~30s，且采样后即关——已基本最优。
- 可与 App 端“舒适/极速/省电”三模式（见 MEMORY：KeyGo 重连三模式）联动：省电模式在固件侧也降采样频率、关 LED。

### 6.4 BLE 6.0 展望（本人瞎想，能提升使用体验）

> 以下为**方向性设想**，当前 CH582M 不支持 BLE 6.0，需未来换芯（如 nRF54 系列）才可能落地。

BLE 6.0（2024 年发布）最相关的特性对 KeyGo 的意义：

#### ① Channel Sounding（基于相位的精准测距）——**与 KeyGo 最相关**
- BLE 6.0 引入 **Channel Sounding**，用**相位测距（PBR）+ 往返时间（RTT）** 实现**亚米级 ~ 厘米级**距离测量，替代/超越传统 RSSI 估算。
- **对 KeyGo 的价值**：
  - 当前 App 用 **RSSI 阈值门控自动解锁**（`KeyGo_ProcessStateMachine` 查 RSSI）。RSSI 受多径、遮挡、人体影响大，精度差且**易受中继攻击（relay attack）**——攻击者把信号放大转发即可“伪造近距离”。
  - Channel Sounding 的距离是**物理测距**，无法被简单放大转发欺骗，可做成**“只有手机在车旁 N 厘米内才允许解锁”**的安全距离门控，**安全性与可靠性远超 RSSI**。
  - 这是未来换支持 BLE 6.0 的芯片（nRF54 系列等）的**核心动力之一**。

#### ② 更精细的 Tx 功率控制 / 能效改进
- BLE 6.0 增强了发射功率控制，可在保证链路质量下进一步降低平均功耗——配合 §6.3 超级省电。

#### ③ Decision-Based Advertising Filtering / 增强广播
- 降低主机扫描功耗、减少无效唤醒——对“后台自动扫描/重连”（Keygo-Foreground 原生服务）的场景利好。

#### ④ PAwR（Periodic Advertising with Responses，5.4 已引入，6.0 生态延续）
- 无连接的大规模双向通信，理论上可用于**车队/多设备群控**（一辆车的多个锁、或共享车队管理），当前 KeyGo 单设备模型暂不需要，但可作架构储备。

#### ⑤ 安全层面
- BLE 6.0 在 LE Secure Connections 基础上强化。对 KeyGo 的绑定/挑战应答体系（`bindKey` + HMAC）是**补充而非替代**——应用层密钥体系仍保留，底层测距安全由 Channel Sounding 提供。

> 小结：BLE 6.0 里**Channel Sounding 是最值得 KeyGo 期待的特性**，它直接解决当前 RSSI 解锁的精度与安全短板；其余多为能效/规模化的渐进改进。是否追 BLE 6.0，取决于换芯成本与“安全距离解锁”需求的优先级。
