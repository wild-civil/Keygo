# KeyGo 加密绑定优化方案设计

> 目标：在量产发布前收口 **「Just Works 配对即可 RSSI 解锁」** 这一安全缺口，同时保留「App 被杀后 OS 自动重连仍能走近解锁」的既有体验。
> 状态：**设计稿（待评审 + 待真机验证 CH582 SMP passkey 支持）**，尚未动代码。
> 关联：现有 KDF/挑战应答见 `code/.../APP/bonding.c` 与 `app/BLE_Key_Go_App/utils/crypto.js`；闸门见 `keygo_core.c:431,442-443`。

---

## 1. 现有机制回顾（衔接基础）

### 1.1 链路层（方案 A）
- `keygo_core.c:442-443` RSSI 自动解锁闸门：
  ```c
  if (!linkDB_State(connHandle, LINK_ENCRYPTED) &&
      !Bonding_IsSessionAuthed(connHandle)) return;
  ```
- `keygo_core.c:431` 未绑定设备直接 `return`（陌生人无法解锁未绑定设备）。
- `bonding.c:124-136` 配对配置：`IO_CAP_NO_INPUT_NO_OUTPUT` + `mitm=0` + `GAPBOND_PERI_BONDING_ENABLED=1` → **Just Works**。

### 1.2 应用层（KDF + 挑战应答 + 签名指令）
- **KDF**：`bindKey[16] = SHA256(utf8(code) || utf8(serial))[0:16]`（`Bonding_DeriveKey` / `crypto.js:deriveBindKey`，两端同源）。`serial` = FF04 的 MAC 大写串（每设备唯一）。
- **BIND**：`Bonding_HandleBindCmd` 校验绑定码（首绑任意码即确立；已绑须匹配 `g_curBindCode`），派生 key 写入信任列表 slot0，置 `s_sessionAuthed=1`。
- **AUTH 挑战应答**：`NONCE` → `AUTH:<HMAC-SHA256(nonce, bindKey) 十六进制>`（`Bonding_HandleAuthResp`）。成功置 `s_sessionAuthed=1`，并用本次 nonce 建 C1 会话盐。
- **C1 签名指令**：`Bonding_VerifySignedCmd` 校验 `HMAC(bindKey, salt||"<body>:<seq>")`，防跨连接/同连接重放。
- 常量时间比较 `ct_eq`、fail-closed（crypto 自测失败拒所有鉴权）、SHA256/HMAC 自测均已落地。

> 结论：应用层鉴权链**已经很强**。缺口只在「链路层身份」——闸门接受的 `LINK_ENCRYPTED` 可由无密码的 Just Works 配对产生。

---

## 2. 当前缺口（Just Works 风险）

**攻击路径**：
1. 车主已 BIND（`s_bondCount>0`），闸门放行条件变为 `LINK_ENCRYPTED || IsSessionAuthed`。
2. 攻击者用任意手机对设备发起 **Just Works 配对**（设备 `mitm=0 / NO_INPUT_NO_OUTPUT`，`WAIT_FOR_REQ` 会接受）。
3. 配对成功 → OS 把该手机记为已 bond → 之后 OS 自动重连并 `LINK_ENCRYPTED=true`。
4. 攻击者走近 → 闸门仅凭「链路已加密」即解锁，**无需知道绑定码**。

`keygo_core.c:440-441` 已自注释此为「已知取舍」。这与「物理手机即钥匙」模型一致，但**量产发布前不应保留**——任何能碰到设备的手机都能悄悄配对后获得永久解锁权。

---

## 3. 设计目标
1. **关闭 Just Works**：`LINK_ENCRYPTED` 只能来自**经绑定码认证的配对**。
2. **保留「App 被杀仍可走近解锁」**：靠 OS 自动加密重连（方案 A 收益），不退回「必须 App 进程活着」。
3. **最小改动**：复用现有 `LINK_ENCRYPTED || IsSessionAuthed` 闸门，**不重写**应用层 KDF/挑战应答。
4. **与现有体系衔接**：绑定码同时作为 SMP Passkey → 一层身份、零额外记忆负担。

---

## 4. 主方案：认证配对（Passkey = 绑定码）

### 4.1 配对配置变更（`bonding.c: Bonding_Init`）
```c
uint8_t ioCap     = GAPBOND_IO_CAP_DISPLAY_ONLY;   // 改自 NO_INPUT_NO_OUTPUT
uint8_t mitm      = 1;                              // 改自 0
// 固定 passkey = 当前有效绑定码（默认 123456 或已载入的自定义码）
uint32_t passcode = Bonding_CodeToPasskey(g_curBindCode, g_curBindCodeLen);
GAPBondMgr_SetParameter(GAPBOND_PERI_PASSCODE, sizeof(uint32_t), &passcode);
```
- 头端声明 `DISPLAY_ONLY` + 设固定 passkey，手机端（Android 多为 `KEYBOARD_DISPLAY`）协商结果 = **Passkey Entry + MITM=1**。
- 头端无需真实屏幕：passkey 由代码提供，手机弹系统框让用户输入。
- 因 `mitm=1`，**SMP 不再回退到 Just Works** → 无码无法配对 → 闸门 `LINK_ENCRYPTED` 仅来自认证配对。
- **闸门逻辑零改动**：`keygo_core.c:442-443` 原样保留，`LINK_ENCRYPTED || IsSessionAuthed` 现在两边都意味着「知道绑定码/密钥」，缺口消除。

### 4.2 绑定码 ↔ 配对码 同步
- **首绑/载入**：`Bonding_Init` 用 `g_curBindCode`（默认或 `Bonding_LoadBindCode` 载入）设 passkey。
- **改码**：`Bonding_HandleSetCodeCmd` 成功更新 `g_curBindCode` 后，立即 `GAPBondMgr_SetParameter(GAPBOND_PERI_PASSCODE, …)` 刷新 passkey，使后续重配对待码一致。
- 两码**强制一致**（同一份 `g_curBindCode`），避免用户记两套。

### 4.3 与应用层 KDF/挑战应答的衔接
- **Passkey 配对** = 链路层身份证明（「你持有绑定码」）。
- **AUTH 挑战应答** = 应用层密钥证明（「你持有由码派生的 bindKey」）+ 建立 C1 会话盐。
- 两者互补，构成**双因子**：配对防蹭链，AUTH/C1 防重放、签指令。现有 `BIND/AUTH/VerifySignedCmd` **全部保留不动**。
- 威胁模型升级：即便攻击者窃听并「重放」某次配对，LTK 由 SMP 用随机 TK 生成（Passkey Entry 下 TK 来自 6 位码 + 随机，MITM 保护），重放无法重建加密链路。

### 4.4 App 侧改动
- `KeygoForegroundModule.java` 的 `createBond` 流程：
  - 调用配对前，用已知绑定码经 `BluetoothDevice.setPin(byte[])`（反射，Android 允许）**静默预填**，避免系统弹框打断；失败则回退系统框手动输入。
  - `BondStateReceiver` 已存在，补充：配对成功 → 继续 BIND/AUTH；配对失败（码错）→ 提示「配对码错误，请输入绑定码」。
- `ble.js _triggerBond` 需把当前 BindModal 的绑定码传给原生 `createBond(code)`。
- `BindModal` 在首绑/重绑时，把用户输入的码同时用于：① SMP passkey ② `BIND` 指令。UX 提示：「系统将请求蓝牙配对，请输入同一绑定码」。

### 4.5 随机数增强（加固，非阻塞）
`Bonding_GenNonce` 当前用 `tmos_rand + xorshift`（已自注「非密码级」）。
- 配对改为认证后，nonce 仅在已加密会话内作抗重放，**泄漏风险已显著降低**。
- 仍建议（可选）：若 CH582 有硬件 RNG 则接入；否则混入 RSSI 抖动/连接事件时间戳做额外熵。优先级低于主方案。

### 4.6 默认码 / 强码（加固）
- 当前默认 `123456`。建议首绑**拒绝弱码**（长度 ≥8、且不与默认相同），或出厂即烧每设备唯一码 + 机身二维码标签（O5 待定）。
- `Bonding_HandleBindCmd` 首绑分支加长度/强度校验即可，改动局部。

## 5. 替代方案对比

| 方案 | 能否关 Just Works | 保留 App 被杀解锁 | 复杂度 | 结论 |
|---|---|---|---|---|
| **A. 认证配对（Passkey=绑定码）** | ✅ | ✅ | 中（改配对配置 + App 预填 pin） | **推荐** |
| B. 去掉 `LINK_ENCRYPTED`，仅 `IsSessionAuthed` | ✅ | ❌（退回方案A 修正前痛点） | 低 | 放弃「App 被杀解锁」收益，不取 |
| C. LESC + Numeric Comparison | ✅ | ✅ | — | ❌ 无头设备无显示/确认，不可行 |
| D. OOB（NFC/QR 带外传密钥） | ✅ | ✅ | 高（需额外硬件/流程） | 超当前范围，可作远期 |
| E. 双因子（A + 现有 AUTH） | ✅ | ✅ | = 方案 A 本身 | 即本方案 |

## 7. 风险与验证

- **⚠ 关键前提**：需真机验证 **CH582/WCH BLE 栈是否真正支持 `DISPLAY_ONLY + 固定 passcode` 的 Passkey Entry 且 `mitm=1` 不回退 Just Works**。若栈不支持，回退到方案 B（仅 `IsSessionAuthed`，接受 App 被杀不解锁）或方案 D。
- 验证用例：
  1. 陌生人手机**不输入码**直接配对 → 应**失败**（无 `LINK_ENCRYPTED`）。
  2. 车主手机输入正确绑定码配对 → 成功 → 走近自动解锁（App 杀掉也行）。
  3. 改绑定码后重配 → 须用**新码**才能配对。
  4. 回归：BIND/AUTH/C1 指令签名、30s 超时强断、DoS 仍正常。
