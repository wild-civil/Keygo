# 多设备管理与已知设备展示方案（计划文档）

> 状态：**P0 已落地（v3.36.3-fix5，已真机验证通过）**；② 列表化 / ③ 已配对徽章 / ④ 默认设备 待讨论或按需展开
> 关联分支：`codebuddy`（2026-07-22 起）
> 背景：当前 App 仅以单值 `ble_last_device_id` 记住「最后连过的那台设备」，多 KeyGo 设备（如汽车 + 电瓶车）场景下体验不足。本文分析四类优化（P0～P3）并给出实施结论。

---

## 0. 现状回顾（为什么需要优化）

| 能力 | 现状 | 文件/代码 |
| --- | --- | --- |
| 扫描列表 | 天然支持多设备，`v-for="device in bleStore.devices"` 列出范围内所有 KeyGo，点哪个连哪个 | `pages/index/index.vue:195` |
| 「重新连接」卡片 | 仅认**单值** `knownDeviceId`（= 最后连过的设备），多设备只显示最近一台 | `stores/ble.js:339` getter |
| 显示名 | 全部 `KeyGo-XXXXXX`（MAC 末 6 位），多设备无法区分 | `stores/ble.js:344` `knownDeviceName` |
| 连接态 | 单设备，`this.deviceId` / `this.connected` 同刻只连一台 | — |
| 自动重连 | 只读 `lastDeviceId`（最后活跃设备） | `stores/ble.js` 多处 |

结论：扫描能用，但**「认得出、一键切、辨已配对」三件事在多设备下都是缺口**。

## ① P0 设备自定义名称（统一为单一命名源）—— 采纳 ✅ 已落地 (v3.36.3-fix5)

### 决策背景：为什么从"平行别名"回退到"单一 `customDeviceName`"

首版 P0 方案（2026-07-22）曾实现过一套**平行别名**体系：扫描列表/重连卡/连接态各加 `✎` 改名按钮、显示绿色「已别名」徽章、store 里新增按 MAC 索引的 `_aliases` / `setDeviceAlias` / `deviceAlias`。

落地后讨论中发现**它与既有 `customDeviceName`（v3.8「连接后自定义名称」）构成了两条平行的命名通道**：

| 维度 | 平行别名（实验期） | 既有 `customDeviceName` |
| --- | --- | --- |
| 入口 | 扫描列表/重连卡/连接态的 `✎` | 连接页「设备管理 > 设备名称 ✍」弹窗 |
| 存储 key | `ble_aliases`（按 MAC 索引） | `ble_device_names`（按 SN 索引） |
| 改名门槛 | 扫到即可改 | 已连接即可改（原设定） |
| 固件同步 | 刻意不写回 | 可选写回 d2（`NAME:xxx`） |

**冲突点**：同一个设备可能既被 `✎` 设成「我的汽车」、又被原弹窗设成「粤B·12345」，屏幕只显示别名（优先级更高），老名字像"丢了"，普通用户分不清该用哪个入口。两条通道、两套 key、两种语义，纯属画蛇添足。

**用户决策（2026-07-22 实测确认）**：
1. 不新增平行别名，**回归原有 `customDeviceName` 作为唯一用户可改名称**（它本就是"为设备命名"的正统入口：连接/绑定后可改、按 SN 存、可写回固件让另一台手机也看到同一名字，天然就承担多设备区分）。
2. 保留绿色徽章的观感，但文案「已别名」→「**已命名**」。
3. 改名门槛改为「**绑定后才能改**」，符合"绑定后才归属该设备"的语义。
4. 已知设备（重连卡）的展示名跟着连接页的 `customDeviceName` 走。

### 命名体系（最终版）

| 名称 | 来源 | 用户可改 | 存储 | 说明 |
| --- | --- | --- | --- | --- |
| **用户自定义名称 `customDeviceName`** | App 内"设置设备名称"弹窗（绑定后） | ✅ | 本地 `ble_device_names`(按 SN) + **按 MAC 副本 `ble_device_custom_names`** | 唯一命名源，多设备一眼区分 |
| 程序生成名 `KeyGo-XXXXXX` | App 按 MAC 末 6 位算 | ❌ | 不存储 | 兜底默认名 |
| 固件广播名 `device.name` | 固件侧 | ❌ | 设备端 | 自定义名不写回固件 |

**显示优先级**：`customDeviceName` ＞ `生成名 KeyGo-XXXXXX` ＞ `原始广播名`。
即：设了自定义名就显示它；没设回退 `KeyGo-XXXXXX`；扫描列表里没自定义名时显示设备广播名。

### 实施要点
- **单一入口**：保留原 index.vue「设备管理 > 设备名称 ✍」弹窗（`showNameDialog` → `setDeviceName`）。实验期的所有 `✎` 平行改名入口全部删除，避免两条命名通道混乱。
- **改名门槛**：须 `isBound`（**绑定后才能改**）；未绑定提示"请先绑定设备后再设置名称"。
- **断连也能显示**（本次关键补强）：原版 `knownDeviceName` 在断连场景只会显示 `KeyGo-XXXXXX`，因为 `customDeviceName` 是连接后按 SN 载入、断连取不到。补强后：`setDeviceName` 与 `_resolveDeviceName` 恢复名称时，**额外按 MAC 持久化一份**（`ble_device_custom_names`），让扫描列表 / 重连卡在断连状态下也能显示自定义名。
- **「已命名」徽章**：`customDeviceName` 已设（非默认）的设备，在扫描列表 / 重连卡 / 连接态设备名条显示绿色「已命名」徽章。

### 逐文件改动清单（v3.36.3-fix5）

**`stores/ble.js`**
- 删除状态 `_aliases`（原 `ble_aliases`）及 getter `deviceAlias(mac)`、action `setDeviceAlias(mac,name)`。
- 新增状态 `_customNamesByMac`（读 `ble_device_custom_names`）+ getter `customNameForMac(mac)`（返回函数供模板传参）+ 辅助 `_seedCustomNameByMac(mac,name)`。
- `knownDeviceName`：改为「`customNameForMac` > 生成名 `KeyGo-XXXXXX`」。
- `connectedDisplayName`：改为「`customDeviceName` > `deviceName`」。
- `setDeviceName(name, syncToDevice)`：设置/恢复 `customDeviceName` 后调用 `_seedCustomNameByMac(deviceId, name)` 按 MAC 落盘。
- `_resolveDeviceName(sn)`：从本地/SN 恢复或首次从固件 d2 记录名称时，同样 `_seedCustomNameByMac` 同步副本。

**`pages/index/index.vue`**
- 重连卡、扫描列表项：删除 `✎` 按钮与 `renameDevice`；徽章文案「已别名」→「已命名」，条件改为 `customNameForMac(mac)`。
- 扫描列表展示函数 `deviceDisplayName(device)`：改为「`customNameForMac(deviceId)` > 设备广播名 `device.name`」。
- `showNameDialog()`：新增 `isBound` 门控（未绑定提示"请先绑定设备后再设置名称"），再开 `pwModal.mode='setName'`。
- 样式：删除 `.rename-btn` / `.reconnect-name-row`，保留 `.device-alias-tag`（「已命名」徽章）。

**`pages/control/control.vue`**
- 重连块、连接态设备名条：删除 `✎` 与 `renameDevice`；加「已命名」徽章（重连块用 `customNameForMac(knownDeviceId)`，连接态用 `customDeviceName`）。
- 样式：删除 `.rename-btn` / `.reconnect-name-row`，保留 `.device-alias-tag`。

### 实现要点（getter / 状态总览）
- Getter：`customNameForMac(mac)`、`knownDeviceName`、`connectedDisplayName`。
- 状态：`_customNamesByMac`；辅助 `_seedCustomNameByMac(mac,name)`。
- 展示点：扫描列表 `deviceDisplayName(device)`（customNameForMac 优先）、重连卡片（index/control）用 `knownDeviceName`、连接态顶部设备名条用 `connectedDisplayName`。

## ② 列表化（多设备都显示 vs 只记最后一台）—— 分析 + 推荐按需展开

### 问题
当前 `knownDeviceId` 是单值，只认最后一台。多设备用户断连后「重新连接」卡片永远只显示最近那台，另一台只能去扫描列表手动找。

### 三种方案对比

| 方案 | 行为 | 成本 | 评价 |
| --- | --- | --- | --- |
| **A 最小（现状）** | 保持单值，多设备靠扫描列表切 | 0 | 多设备每次要手动扫，体验差 |
| **B 按需展开（推荐）** | 维护 `knownDevices` 集合（所有连过的 MAC + 别名 + lastConnectedAt）。已知设备 = 1 台 → 单卡（现状）；≥2 台且都未连 → 卡片展开为可滚动列表，每项含别名/MAC/「重新连接」按钮 | 中 | 单设备退化为 A（零额外成本），多设备才展开，兼顾简单与能力 |
| **C 全列表** | 始终以列表呈现 | 中高 | 单设备也变列表，略重，没必要 |

### 结论（写入文档，待落地）
- **先落地 P0 + P2**（自定义名称 + 已配对标记），这两项是单设备也受益、成本最低的。
- **P1 列表化标记为「可选增强」**：当用户确有 2+ 台 KeyGo 时再上，采用方案 B。
- 无论 A/B，底层**始终保留单值 `lastDeviceId` 作为自动重连依据**（见 ④），列表化只影响「重新连接」卡片的展示与手动点选，不动自动重连路径。

## ③ P2 扫描列表标记「已配对」 —— 采纳 ✅

### 判定「已配对 / 已绑定」
- **近似判定（推荐先做）**：以本地 `knownDevices` 集合中存在的 MAC 为准 = 本机曾成功连接/绑定过的设备即视为「已配对」。
- 更严格做法（读取 OS BLE 绑定列表）成本高、跨平台差异大，暂不做；本地集合已足够覆盖 99% 场景（陌生人设备不在集合里）。

### 实施要点
- 扫描列表项：MAC 命中 `knownDevices` → 显示绿色「已配对」徽章；未命中 → 无徽章（或灰字「陌生设备」）。
- 目的：**一眼区分自有设备 vs 他人设备，防误连别人车**。
- 与 ① 联动：徽章设备同时用自定义名称显示。

---

## ④ P3 多设备自动重连策略 —— 维持现状，默认设备功能暂不做

### 现状（用户认可）
自动重连只读 `lastDeviceId`（最后活跃的那台）。多台在范围内也连最后活跃设备——**用户确认此行为即可**。

### 「设为默认设备」功能 —— 规划中，暂未实现 ⏸
- 设想：让用户把某台标记为「默认」，自动重连优先它。
- **未做原因（已在文档标明）**：需新增设置 UI 入口（选择/置顶），**用户可能嫌繁琐**；且多设备同场概率低，当前「连最后活跃」已满足需求。
- 后续若用户反馈多设备同场连错，再评估启用。

## 实施优先级（结论）

| 优先级 | 项 | 状态 | 说明 |
| --- | --- | --- | --- |
| **P0** | 设备自定义名称（统一为单一 `customDeviceName` 源） | ✅ 已实现 (v3.36.3-fix5) | 绑定后才能改；断连也能显示；绿色「已命名」徽章；无平行别名通道 |
| **P2** | 扫描列表「已配对」徽章 | ✅ 采纳，待实现 | 与 P0 联动，防误连 |
| **P1** | 「重新连接」列表化（方案 B） | ⏸ 可选增强 | 用户确有 2+ 台时再上 |
| **P3** | 默认设备置顶 | ⏸ 暂不做 | 用户嫌繁琐 + 多设备同场概率低 |
