# 多设备管理与已知设备展示方案（计划文档）

> 状态：**① P0 + ② 列表化 + ③ 已配对徽章 + ④ 默认设备置顶 均已落地（2026-07-23）**；均为纯前端改动，无原生/固件改动
> 关联分支：`codebuddy`（2026-07-22 起）
> 背景：当前 App 仅以单值 `ble_last_device_id` 记住「最后连过的那台设备」，多 KeyGo 设备（如汽车 + 电瓶车）场景下体验不足。本文分析四类优化（P0～P3）并给出实施结论。

---

## 0. 现状回顾（为什么需要优化）

| 能力 | 现状 | 文件/代码 |
| --- | --- | --- |
| 扫描列表 | 天然支持多设备，`v-for="device in bleStore.devices"` 列出范围内所有 KeyGo，点哪个连哪个 | `pages/index/index.vue:195` |
| 「重新连接」卡片 | 仅认**单值** `knownDeviceId`（= 最后连过的设备），多设备只显示最近一台 | `stores/ble.js:339` getter |
| 显示名 | 全部 `KeyGo-XXXXXX`（MAC 末 6 位），多设备无法区分 | `stores/ble.js:344` `knownDeviceName` |
| 连接态 | 单设备，`this.deviceId` / `this.connected` 同刻只连一台 | — |
| 自动重连 | 只读 `lastDeviceId`（最后活跃设备） | `stores/ble.js` 多处 |

结论：扫描能用，但**「认得出、一键切、辨已配对」三件事在多设备下都是缺口**。

## ① P0 设备自定义名称（统一为单一命名源）—— 采纳 ✅ 已落地 (v3.36.3-fix5)

### 决策背景：为什么从"平行别名"回退到"单一 `customDeviceName`"

首版 P0 方案（2026-07-22）曾实现过一套**平行别名**体系：扫描列表/重连卡/连接态各加 `✎` 改名按钮、显示绿色「已别名」徽章、store 里新增按 MAC 索引的 `_aliases` / `setDeviceAlias` / `deviceAlias`。

落地后讨论中发现**它与既有 `customDeviceName`（v3.8「连接后自定义名称」）构成了两条平行的命名通道**：

| 维度 | 平行别名（实验期） | 既有 `customDeviceName` |
| --- | --- | --- |
| 入口 | 扫描列表/重连卡/连接态的 `✎` | 连接页「设备管理 > 设备名称 ✍」弹窗 |
| 存储 key | `ble_aliases`（按 MAC 索引） | `ble_device_names`（按 SN 索引） |
| 改名门槛 | 扫到即可改 | 已连接即可改（原设定） |
| 固件同步 | 刻意不写回 | 可选写回 d2（`NAME:xxx`） |

**冲突点**：同一个设备可能既被 `✎` 设成「我的汽车」、又被原弹窗设成「粤B·12345」，屏幕只显示别名（优先级更高），老名字像"丢了"，普通用户分不清该用哪个入口。两条通道、两套 key、两种语义，纯属画蛇添足。

**用户决策（2026-07-22 实测确认）**：
1. 不新增平行别名，**回归原有 `customDeviceName` 作为唯一用户可改名称**（它本就是"为设备命名"的正统入口：连接/绑定后可改、按 SN 存、可写回固件让另一台手机也看到同一名字，天然就承担多设备区分）。
2. 保留绿色徽章的观感，但文案「已别名」→「**已命名**」。
3. 改名门槛改为「**绑定后才能改**」，符合"绑定后才归属该设备"的语义。
4. 已知设备（重连卡）的展示名跟着连接页的 `customDeviceName` 走。

### 命名体系（最终版）

| 名称 | 来源 | 用户可改 | 存储 | 说明 |
| --- | --- | --- | --- | --- |
| **用户自定义名称 `customDeviceName`** | App 内"设置设备名称"弹窗（绑定后） | ✅ | 本地 `ble_device_names`(按 SN) + **按 MAC 副本 `ble_device_custom_names`** | 唯一命名源，多设备一眼区分 |
| 程序生成名 `KeyGo-XXXXXX` | App 按 MAC 末 6 位算 | ❌ | 不存储 | 兜底默认名 |
| 固件广播名 `device.name` | 固件侧 | ❌ | 设备端 | 自定义名不写回固件 |

**显示优先级**：`customDeviceName` ＞ `生成名 KeyGo-XXXXXX` ＞ `原始广播名`。
即：设了自定义名就显示它；没设回退 `KeyGo-XXXXXX`；扫描列表里没自定义名时显示设备广播名。

### 实施要点
- **单一入口**：保留原 index.vue「设备管理 > 设备名称 ✍」弹窗（`showNameDialog` → `setDeviceName`）。实验期的所有 `✎` 平行改名入口全部删除，避免两条命名通道混乱。
- **改名门槛**：须 `isBound`（**绑定后才能改**）；未绑定提示"请先绑定设备后再设置名称"。
- **断连也能显示**（本次关键补强）：原版 `knownDeviceName` 在断连场景只会显示 `KeyGo-XXXXXX`，因为 `customDeviceName` 是连接后按 SN 载入、断连取不到。补强后：`setDeviceName` 与 `_resolveDeviceName` 恢复名称时，**额外按 MAC 持久化一份**（`ble_device_custom_names`），让扫描列表 / 重连卡在断连状态下也能显示自定义名。
- **「已命名」徽章**：`customDeviceName` 已设（非默认）的设备，在扫描列表 / 重连卡 / 连接态设备名条显示绿色「已命名」徽章。

### 逐文件改动清单（v3.36.3-fix5）

**`stores/ble.js`**
- 删除状态 `_aliases`（原 `ble_aliases`）及 getter `deviceAlias(mac)`、action `setDeviceAlias(mac,name)`。
- 新增状态 `_customNamesByMac`（读 `ble_device_custom_names`）+ getter `customNameForMac(mac)`（返回函数供模板传参）+ 辅助 `_seedCustomNameByMac(mac,name)`。
- `knownDeviceName`：改为「`customNameForMac` > 生成名 `KeyGo-XXXXXX`」。
- `connectedDisplayName`：改为「`customDeviceName` > `deviceName`」。
- `setDeviceName(name, syncToDevice)`：设置/恢复 `customDeviceName` 后调用 `_seedCustomNameByMac(deviceId, name)` 按 MAC 落盘。
- `_resolveDeviceName(sn)`：从本地/SN 恢复或首次从固件 d2 记录名称时，同样 `_seedCustomNameByMac` 同步副本。

**`pages/index/index.vue`**
- 重连卡、扫描列表项：删除 `✎` 按钮与 `renameDevice`；徽章文案「已别名」→「已命名」，条件改为 `customNameForMac(mac)`。
- 扫描列表展示函数 `deviceDisplayName(device)`：改为「`customNameForMac(deviceId)` > 设备广播名 `device.name`」。
- `showNameDialog()`：新增 `isBound` 门控（未绑定提示"请先绑定设备后再设置名称"），再开 `pwModal.mode='setName'`。
- 样式：删除 `.rename-btn` / `.reconnect-name-row`，保留 `.device-alias-tag`（「已命名」徽章）。

**`pages/control/control.vue`**
- 重连块、连接态设备名条：删除 `✎` 与 `renameDevice`；加「已命名」徽章（重连块用 `customNameForMac(knownDeviceId)`，连接态用 `customDeviceName`）。
- 样式：删除 `.rename-btn` / `.reconnect-name-row`，保留 `.device-alias-tag`。

### 实现要点（getter / 状态总览）
- Getter：`customNameForMac(mac)`、`knownDeviceName`、`connectedDisplayName`。
- 状态：`_customNamesByMac`；辅助 `_seedCustomNameByMac(mac,name)`。
- 展示点：扫描列表 `deviceDisplayName(device)`（customNameForMac 优先）、重连卡片（index/control）用 `knownDeviceName`、连接态顶部设备名条用 `connectedDisplayName`。

## ② 列表化（多设备都显示 vs 只记最后一台）—— 分析 + 推荐按需展开

### 问题
当前 `knownDeviceId` 是单值，只认最后一台。多设备用户断连后「重新连接」卡片永远只显示最近那台，另一台只能去扫描列表手动找。

### 三种方案对比

| 方案 | 行为 | 成本 | 评价 |
| --- | --- | --- | --- |
| **A 最小（现状）** | 保持单值，多设备靠扫描列表切 | 0 | 多设备每次要手动扫，体验差 |
| **B 按需展开（推荐）** | 维护 `knownDevices` 集合（所有连过的 MAC + 别名 + lastConnectedAt）。已知设备 = 1 台 → 单卡（现状）；≥2 台且都未连 → 卡片展开为可滚动列表，每项含别名/MAC/「重新连接」按钮 | 中 | 单设备退化为 A（零额外成本），多设备才展开，兼顾简单与能力 |
| **C 全列表** | 始终以列表呈现 | 中高 | 单设备也变列表，略重，没必要 |

### 结论（写入文档，待落地）
- **先落地 P0 + P2**（自定义名称 + 已配对标记），这两项是单设备也受益、成本最低的。
- **P1 列表化标记为「可选增强」**：当用户确有 2+ 台 KeyGo 时再上，采用方案 B。
- 无论 A/B，底层**始终保留单值 `lastDeviceId` 作为自动重连依据**（见 ④），列表化只影响「重新连接」卡片的展示与手动点选，不动自动重连路径。

## ③ P2 扫描列表标记「已配对」 —— 采纳 ✅

### 判定「已配对 / 已绑定」
- **近似判定（推荐先做）**：以本地 `knownDevices` 集合中存在的 MAC 为准 = 本机曾成功连接/绑定过的设备即视为「已配对」。
- 更严格做法（读取 OS BLE 绑定列表）成本高、跨平台差异大，暂不做；本地集合已足够覆盖 99% 场景（陌生人设备不在集合里）。

### 实施要点
- 扫描列表项：MAC 命中 `knownDevices` → 显示绿色「已配对」徽章；未命中 → 无徽章（或灰字「陌生设备」）。
- 目的：**一眼区分自有设备 vs 他人设备，防误连别人车**。
- 与 ① 联动：徽章设备同时用自定义名称显示。

## ④ P3 多设备自动重连策略 —— 维持现状，默认设备置顶已落地（列表层）

### 现状（用户认可）
自动重连只读 `lastDeviceId`（最后活跃的那台）。多台在范围内也连最后活跃设备——**用户确认此行为即可**。

### 「设为默认设备」功能 —— 已落地（列表置顶，自动重连不变）✅
- **2026-07-23 实现**：在「重新连接」卡片（index/control 双页）提供「默认」按钮，可将某台标记为默认；
  标记后 `knownDevicesList` 排序时**默认设备置顶**（列表层 Pinned-to-top）。
- **刻意未改自动重连**：自动重连仍只读 `lastDeviceId`（用户已认可的行为），默认仅影响手动重连列表的顺序，
  避免引入"用户觉得连错设备"的回归风险。
- **单设备退化**：只有 1 台时列表退化为单卡，「默认」按钮不显示（无意义），与现状完全一致。
- **⚠️ 「已知设备」框出现时机（2026-07-23 记录）**：该框仅在 `未连接` 且 `knownDevices` 非空时显示（index 重连卡片 / control 重连块均受此门控）。
  - 多设备隐患：已连设备 A 时框隐藏 → 用户想从 A 切到 B 没有入口。若需「已连 A 时切换 B」，需新增已连接态切换入口（如在控制页顶部车辆卡加下拉/列表），否则多设备只能先断开再连。
  - 另一时机点：首次安装从未连接过 → 集合为空 → 无框（符合预期）；首次成功连接并断开后才出现。建议保持"仅未连接态出现"，避免与自动重连状态卡(state-text)视觉冲突。

## 已评估 / 已否决的优化项（2026-07-23 复盘，未落地）

> 以下为针对本方案衍生优化点的代码级评估。均**未实现**，结论供后续决策。

### A. 多设备「已连接态切换入口」（评估：暂不做）

**代码事实**：重连 UI 两处都受 `!connected` 门控——
- `index.vue:171` 重连卡片：`v-if="!bleStore.connected && bleStore.knownDevicesList.length"`
- `control.vue:4` 外层 `conn-warning`：`v-if="!bleStore.connected"`，内部 `:8` 重连块：`v-if="bleStore.knownDevicesList.length"`

即已连设备 A 时整块不可见，**确实没有切到 B 的入口**，只能先手动断开再连。缺口为真。

**必要性**：`knownDevices` 按手机记录，一人手机出现 2+ 台 = 同一用户拥有 2+ 辆车/设备（小众）。且自动重连认 `lastDeviceId`，日常靠近哪台自动连哪台，多设备场景大多无需手动切。"已连 A 想切 B"仅在同时带两把钥匙且当前连错时发生，概率极低。

**结论：低优先级，暂不做。** 若硬做，最小成本是在 `control.vue` 车辆大卡加切换下拉（`disconnect()`+`connect()`），但引入误触断开风险，收益不抵成本。等真实用户反馈再上。

### B. handleReconnect / handleSetDefault 去重（评估：暂不去重）

**两页实现对比**（index.vue `handleReconnect` 621–634 / `handleSetDefault` 636–644；control.vue 291–303 / 305–313）：
- `handleReconnect(mac)`：两页逻辑逐字相同——`showLoading` → `await bleStore.connect(mac, bleStore._resolveFactoryName(mac))` → 成功/失败 `hideLoading`+`toast`。
- `handleSetDefault(mac)`：两页**完全一致**——`await bleStore.setDefaultDevice(mac)` + `toast.success('已设为默认设备')`。
- **唯一差异**：index 的 `handleReconnect` 在 `connected` 成功后多一行 `bleStore.devices = []`（清扫描列表缓存；control 页无扫描列表故不需要）。但 `bleStore.devices` 是 store 公共 state，在 control 里清空无害，抽共享函数时保留该行两页都安全。

**优劣**：去重（抽 `composables/useKnownDevices.js`）优点=重连逻辑只写一处；缺点=多一个文件、函数仅 8 行且差异 1 行，属可接受的小重复，过度抽象反而难维护。

**结论：收益低，暂不去重。** 仅当第 3 个页面复用、或重连逻辑变复杂（加超时/失败引导）时再抽。

### C. knownDevicesList getter "每渲染读 storage"（评估：伪优化，忽略）

**纠正前提**（此前误记为"每渲染重算 storage"）：
- `customNameForMac(mac)`（`ble.js:405`）：读 `state._customNamesByMac[key]`——**内存对象**，store 初始化时一次性从 storage 载入，非每调用读 storage。
- `_resolveFactoryName(mac)`（`ble.js:2436`）：优先读 `this._advertisedNames[macKey]`（**内存缓存**，初始化时载入），否则 `_factoryNameForMac` 纯 MAC 计算，无 I/O。
- `knownDevicesList` 是 **Pinia getter = computed**，依赖（`knownDevices`/`defaultDeviceId`/`_customNamesByMac`/`_advertisedNames`）不变时不重算，更不会被"每次渲染"触发。

**影响**：真正成本 = O(N) 字符串拼接，N = 本机连过设备数（车锁场景个位数）。无 storage I/O、无网络。即便 N 涨到 50 也是微秒级。

**结论：无需改，不会造成任何实际问题。** 若强行"优化"（结果存 ref）属过度设计。

### D. 配置 uc/lc/阈值 与 dormant 持久化（评估：不要做）

#### D1 解锁/锁车阈值写 DataFlash（评估：不要做，现有设计正确）
**固件侧**（`keygo_core.c` `KeyGo_ParseConfig` `:1267`）：per-phone 项 `unlock/lock/uc/lc/interval/dlock` **仅存 RAM**；只有设备级 `cooldown_ms` 变更才调 `KeyGo_SaveConfig()`（`:1568`，`EEPROM_ERASE(KEYGO_CFG_ADDR,256)` 整页擦写，罕见）。
**App 侧兜底**：连接成功 `_syncConfigToDevice`（`ble.js:2289`）重推阈值；`_persistConfig()`（`:2230`）存手机本地 `ble_config_v1_{SN}`，App 重启即恢复。
**结论**：阈值不写 DataFlash → 无磨损；靠"手机存储 + 连接重推"补 RAM 易失。强行把 uc/lc 也写 DataFlash（早期 roadmap 一条）反而引入无谓擦写，**有害无益**。保持现状。

#### D2 dormant 跨重启持久化（评估：不做）
`reconnectMode='dormant'` 有**两种不同语义**的赋值点：
1. 用户手动断开（`disconnect()` `ble.js:2697`）；
2. 固件踢人 BIND:TIMEOUT（`ble.js:3651`）——被动安全抑制。

**现状**：第 2 种**已持久化**（`:3652-3656` 写 `keygo_unbound_kicked`，`_shouldAutoReconnect` 据此拦截，手动 `connect()` 才清除，是 2026-07-12 专门修的坑）；第 1 种**不持久化**，App 重启后 `reconnectMode` 默认回 `idle`（`ble.js:242`）→ 自动重连。
**结论**：不做"dormant 跨重启持久化"。BIND:TIMEOUT 抑制已持久化且正确；用户手动断开不持久化对车锁更合理（手机重启后仍希望自动连回自己的车）。若未来要"记住用户不想连"，应新增**独立标志**（如 `keygo_user_manual_offline`）与 `keygo_unbound_kicked` 分开，而非复用 `dormant`。

### E. 命名相关两个讨论点（评估：E1 非 bug / E2 建议做）

#### E1 自定义名 == 出厂名 会怎样（评估：非 bug，可选规范化）
- **主路径已防重**：`_formatDisplayName`（`ble.js:2476`）`if (custom && factory && custom !== factory) return custom + ' ( ' + factory + ' )'`——自定义名与出厂名**完全相同**时走 `return custom || factory`，即只显示 `KeyGo-XXXXXX`，**不会出现 `KeyGo-XXXXXX ( KeyGo-XXXXXX )` 冗余**。
- **边缘情况**：若用户输入 `keygo-5e3d0c`（大小写不同）或 `KeyGo-5E3D0C `（含尾空格），因 `!==` 为大小写/空格敏感比较，会显示冗余 `keygo-5e3d0c ( KeyGo-5E3D0C )`。可加 `trim()` + 大小写不敏感归一化比较消除。
- **副作用**：即便自定义名==出厂名，`customNameForMac` 返回非空 → 扫描列表/重连卡仍显示绿色「已命名」徽章，用户可能困惑"明明是默认名为何标已命名"。
- **结论**：非功能 bug，优先级低。可选：① 比较前 `trim`+归一化；② 将"自定义名==出厂名"视为"未命名"（跳过徽章 / 或自动清除）。

#### E2 是否需要「恢复原名」选项（评估：**已落地 2026-07-23**）

- **现状缺口（修复前）**：UI **无任何清除入口**。`showNameDialog`/`handleSetName`（`index.vue:722`/`744`）只支持"改名"，`handleSetName` 在 `!name` 时 `toast.info('请输入设备名称')` 直接拒绝（`:746`）。即命名是**单向**的——能改成别的，不能撤销回出厂默认。
- **双存储坑（关键，为什么必须改两处）**：自定义名同时存两份、且由**不同界面读取**：

  | 存储 | key / state | 谁写 | 谁读（界面） |
  |------|-------------|------|--------------|
  | **SN 副本** | `ble_device_names` → `customDeviceName` | `setDeviceName` 直接写 | **已连接**态：`index.vue:29` 的 `custom-name-display`、`control.vue:50` 的 `car-identity`（走 `controlTopName`→`customDeviceName`） |
  | **MAC 副本** | `ble_device_custom_names` → `_customNamesByMac` | `setDeviceName` 经 `_seedCustomNameByMac` 写 | **断连**态：`knownDevicesList` getter `ble.js:364-369`（`customName` 徽章源 / `displayName` 名字源）、扫描列表 `index.vue:645 deviceDisplayName` |

  ⚠️ **只清一份会"连着是原名、断开变旧名"**：即便 `setDeviceName('')` 把 `customDeviceName=''`（SN 副本清了，已连接态回落出厂名 ✓），但旧 `_seedCustomNameByMac(deviceId,'')` 因 `if (!mac || !name) return`（`ble.js:2380`）**提前返回 → MAC 副本不删** → 断连后的重连卡 / 扫描列表仍显旧名 + 绿色「已命名」徽章（✗）。这正是"恢复不全"的根因——**两份存储必须同时清，不能只清 SN 那份**。

- **落地方案（改两处 + UI 入口）**：
  1. **`_seedCustomNameByMac`（`ble.js:2379`）改为"空名即删"**：去掉 `!name` 提前 return；`name` 非空→`next[key]=name`，`name` 为空→`delete next[key]`，再统一 `setStorageSync`。这样 `setDeviceName('')` 才能把 MAC 副本真正删掉。
  2. **`setDeviceName('')` 支持恢复（清空两份本地存储）**：`customDeviceName=''` 已清 SN 内存态；SN 持久化处分空名分支——`delete _deviceNames[serialNumber]`（而非存 `{name:''}` 留空串），再 `_saveDeviceNames()`。`syncToDevice` 默认 `false`（仅清本机；需要连设备 d2 也复位时由调用方传 `true`）。
  3. **`handleSetName`（`index.vue:744`）允许空名 = 恢复**：`!name` 不再拒绝，改为 `await bleStore.setDeviceName(name)`（`name=''` 即恢复），toast 文案区分"已恢复默认名称"/"设备名称已更新"。
  4. **入口设计（不单独加框）**：恢复默认名**不**另设「恢复默认名称」按钮，只保留「设备名称」一行——在名称弹窗里**清空输入框并保存**即恢复；弹窗 hint 在已命名时提示"清空输入框并保存可恢复为出厂名称"。避免冗余二次确认，命名本就 opt-in、可撤销即可。

- **正确性验证（三处展示均回落出厂名）**：
  - 已连接：`customDeviceName=''` → `index.vue:29` 隐藏自定义名条；`controlTopName`（`ble.js:435`）回退 `connectedDisplayName` → 出厂名。✓
  - 断连重连卡：`knownDevicesList` getter 依赖 `_customNamesByMac` 变化重算 → `customNameForMac(mac)=''` → 徽章 `v-if="d.customName"` 不显；`displayName: custom||factory` = 出厂名。✓
  - 扫描列表：`deviceDisplayName` 走 `customNameForMac` = '' → 出厂名，无「已命名」徽章。✓
  - 重启后：`_restoreDeviceName`（`ble.js:2398`）`if (entry && entry.name)` 因 SN 条目已删（无 `name`）跳过 → `customDeviceName` 保持默认；MAC 副本已删 → 两端均回落。✓

- **结论：已落地**。命名变为可逆（改名 / 恢复默认）；实现重点在"同时清理 SN + MAC 两份存储"，仅靠 `setDeviceName('')` 不清 `_seedCustomNameByMac` 的空名分支会留下断连态旧名残留。E1（`自定义名==出厂名` 的冗余/徽章）视精力顺手规范化比较即可。
