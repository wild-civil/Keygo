## 一、绑定握手协议：两端一致、正确 ✅

逐项核对 App 与固件，无协议层 bug：

| 环节 | App 侧 | 固件侧 | 一致性 |
|---|---|---|---|
| KDF | `deriveBindKey`: `SHA256(utf8(code)‖utf8(serial))[0:16]` | `Bonding_DeriveKey`: 同 `code‖serial` 取前 16 | ✅ |
| HMAC | `HMAC(key=bindKey[16], msg=nonce[16])` | `hmac_sha256(key=bindKey,16, msg=nonce,16)` | ✅ |
| NONCE | 发 `NONCE` → 等 `NONCE:<32hex>` | `Bonding_HandleNonceReq` 回 `NONCE:<32hex>` | ✅ |
| AUTH | 回 `AUTH:<64hex>` → 等 `AUTH:OK` | HMAC 比对后回 `AUTH:OK/FAIL` | ✅ |
| `tmos_memcmp` 语义 | — | 已确认 `TRUE=相等`（`CH58xBLE_ROM.h:2635`），`bonding.c` 比较逻辑正确 | ✅ |
| FF03 解析 | 发 `BIND:code` | `peripheral.c:744` 剥 `BIND:`(5B) 传 code | ✅ |
| 回包送达 | 等 `BIND:OK`(2.5s)+AUTH 兜底 | `KeyGo_SendRawNotify` 延迟队列+重试 | ✅ |

**结论：只要手机烧的是含「延迟发送短报文」的新固件，绑定 handshake 一定能成。**

## 二、绑不上的唯一根因 = 旧固件丢回包（治本已落，须重烧）

- 旧固件在 FF03 写回调内**同步**发 `BIND:OK/NONCE/AUTH`，此时 `Bonding_Save()` 的 Flash 写占总线 → 这些短包被丢。NONCE 同样被丢 → AUTH 握手起不来。
- App 侧一切兜底（等 BIND:OK / AUTH 兜底 / status `bn:1`）都救不了旧固件——因为根因在"写回调同步发"，不在时序。
- **治本修复（已写入 `keygo_core.c` + `peripheral.h` + `peripheral.c`，工作区已就绪，待 MRS 编译烧录）**：`KeyGo_SendRawNotify` 改为写回调内只入队 + 启动 `SBP_DEFERRED_RAW_EVT`，由 `KeyGo_FlushRawNotify` 在 TMOS 任务里（与 status 同可靠通道）发送并带忙窗口重试。

> 验证第一步永远看弹框里的「固件版本」徽标：显示 `3.30.2+` 即新固件已生效；仍 `3.30.0/3.30.1` 即烧录未生效。

## 三、★ 深挖发现的安全大坑（已修复，未提交）

`crypto_sha256.c` 的自测写反了：

```c
// 原（错误）：tmos_memcmp 返回 TRUE=相等，算对时返回非0 → !=0 为真 → 误报失败
if (tmos_memcmp(d, exp_abc, SHA256_DIGEST_LEN) != 0) r |= 1;
if (tmos_memcmp(d, exp_hmac, SHA256_DIGEST_LEN) != 0) r |= 2;
```

后果：`g_cryptoOk`（`bonding.c`）**恒为 0** → `Bonding_HandleAuthResp` 永远走降级分支「设备已绑定即 `AUTH:OK`」→ **任何连上的客户端都能直控车（认证被完全绕过）**。

**修复（2026-07-11，未提交，随下次固件编译一起烧录）**：两行 `!=0` → `==0`（不相等才置错误位）。
- crypto 实现正确时 → `g_cryptoOk=1` → AUTH 走真实 HMAC 校验（安全恢复）；
- crypto 实现仍被破坏时 → 自测正确 FAIL → `g_cryptoOk=0` → 自动降级（仍可用，等同改前）。
- 两端均无回退，纯增益。与 `bonding.c:304/395` 的 `==0` 修复同构、自洽。

> ⚠️ 历史纠正：当日早轮次日志曾误写「自测 `!=0`（不同→置位）是正确用法」，该说法错误；按 `tmos_memcmp` 语义，自测"不相等才报错"必须 `==0`。

## 四、绑定码长度设计讨论（用户待定：A / B / C）

### 现状约束（固件已支持变长）
- `BOND_CODE_MAXLEN = 32`；`SETCODE` 接受 `1..32`；`BIND` 要求「长度 == 当前码长且逐字节相等」（`bonding.c:358`）。
- App `deriveBindKey` 接受任意字符串长度。
- **故「固定 6」还是「6–12」只需改 App 输入校验 + 生成器，协议层零改动。**

### 安全前提（决定性）
KDF 是**单次 SHA256，无慢哈希（PBKDF2/argon2）**。绑定码 **= 全部熵**：
- 攻击者抓一次 `NONCE`+`AUTH` 回包 + 读 `serial`（FF04 现不加密可读）→ 可离线穷举 `SHA256(猜测码‖serial)[0:16]`。
- 6 位数字 = 10⁶ ≈ **20 bit** → 秒破；
- 12 位字母数字 = 62¹² ≈ **72 bit** → 不可破。

### 结论建议（按用途分两档，不要"统一固定 6"）
- **owner 出厂/设置码**：保留 **6 位数字**（贴机身、人手动输入，usability 优先；物理持有即恢复）。
- **生成的临时共享码**：用 **12 位字母数字随机串**（机器生成、App 内复制分享、无人手敲，正好补"无慢哈希"短板）。**不要把临时码也限成 6 位数字。**

### 关键约束：单码模型
当前固件是**单码模型**（仅 `g_curBindCode` 一个槽）。生成临时码 = `SETCODE` **覆盖**当前码；owner 重绑须输入同一串。若想「owner 短码 + guest 长临时码并存」= 需第二码槽 + guest 信任条目（旧 L2 计划，未做）。

### 待用户拍板的三个方案
- **A**：加 App 侧"最小 6 位"校验 + 一个「生成临时随机码」按钮（默认 12 位字母数字、可复制），落进单码顶替当前码。
- **B**：先只统一成「6–12 字母数字」输入框规则，临时码生成以后再说。
- **C**：做真正的双码（owner 短码 + guest 长临时码并存），需扩展固件信任模型，工作量更大。

> **用户当前决策（2026-07-11 晚）**：先以 123456 默认码把绑定验证跑通（验证优先），绑定码长度/临时码生成等优化**后期再做**，暂不选 A/B/C。
