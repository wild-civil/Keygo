## 二、断开原因码（BLE HCI reason）速查

固件 `Disconnected.. Reason:%x` 打印的是 HCI 错误码，常见：

| reason(hex) | 含义 | 典型场景 |
|---|---|---|
| `0x08` | **连接监督超时** | 锁屏/Doze/信道差，**超时设 1s 时高发** → 断连主嫌 |
| `0x13` | 对端主动断开 (remote user terminated) | 手机 App/OS 主动断 |
| `0x16` | 本地主动断开 (local host terminated) | 固件 30s 强断 / TerminateLink |
| `0x3E` | 建立连接失败 | 连接阶段就失败 |
| `0x22` | LMP 响应超时 | 配对/加密协商卡住 |

**判定口诀**：
- 大量 `0x08` → 监督超时太短（1s），验证「提到 4s」。
- `0x16` + App 侧 `存活~30s authed=false` → 命中「未鉴权 30s 强断」。
- `0x13` → 手机侧断的，查 App 重连逻辑/系统蓝牙。

## 三、复现测试流程

### 阶段 0：准备
1. 固件烧录含本轮埋点的构建（注意：改的是 `.c/.h`，MRS 直接编译烧录即可；**非原生插件**，App 无需重做基座）。
2. 串口工具连 CH582M（115200），准备记录 `[DIAG]` / `Reason` 日志。
3. App 打开 DEV 面板日志。
4. 备两台安卓手机 A、B。

### 阶段 1：复现「B 配对后 A 被迫重配」（多手机互踢）
> 目标：用 `snvBonds` 变化实证 LTK 被挤占。

1. 设备**恢复出厂**（隐藏键长按 5s，或 App `UNBIND:1`）。串口应见 `[DIAG][INIT] appOwners=0 snvBonds=0`。
2. **A 手机**：完成绑定（首绑输 123456）→ 确认能解锁/锁车。
   - 若 App 开了「无 App 模式 / usePasskey」（会走 OS 系统配对）：串口应见 `[DIAG][BOND_SAVED] ... snvBonds=1`。
   - 若纯明文 BIND（默认）：`snvBonds` 可能仍为 0（不产生 LTK），此时**互踢与 SNV 无关**，见下方备注。
3. A 断开重连一次 → 确认 App 自动 AUTH 成功（`[DIAG][AUTH]`）、能控车。
4. **B 手机**：绑定 / 系统配对。观察串口每个 `[DIAG]` 的 `snvBonds`：
   - **若 `snvBonds` 从 2→1（或维持 1 但 owner key 变了）** → 实证 A 的 LTK 被淘汰。
5. **切回 A 手机**尝试连接：
   - A 弹「需要重新配对」？→ 命中 SNV 挤占（OS 层）。
   - A 不弹但 App 显示「需重绑」(`devBound=true bound=false`)？→ 命中应用层 slot0 被覆盖（key 身份问题）。

> **备注（关键）**：应用层 `bindKey = SHA256(绑定码 || 设备MAC)`。若 A、B 用**同一绑定码**，派生出的 key **相同** → 即便 slot0 被覆盖，A 仍能 AUTH（key 一样）。所以：
> - 「A 被迫重配」若发生在**纯明文模式**，更可能是 **CH582M 单连接槽**（B 连着时 A 连不上）或 App 重连逻辑，而非密钥问题。
> - 若发生在**系统配对/加密模式**，则是 **SNV LTK 容量**问题（`BLE_SNV_NUM=1`）。
> 阶段 1 的日志正是用来区分这两条路径的。

### 阶段 2：复现「APP 偶发断连」
1. 单手机连接后，记录 `[DIAG] LinkEst` 与约 4s 后的 `[DIAG] ParamUpd` 的 `timeout` 值。
2. 手机**锁屏静置 5~10 分钟**，观察是否周期性断连。
3. 每次断连记录：固件 `Reason:xx` + App `断连: 存活..s authed=..`。
4. 统计 reason 分布：
   - 多为 `0x08` → 进入阶段 3 验证监督超时。
   - 多为 `存活~30s authed=false` + `0x16` → 未鉴权强断，查为何 AUTH 没及时完成。

### 阶段 3：监督超时对照验证（改 1 个宏）
1. `peripheral.h`：`DEFAULT_DESIRED_CONN_TIMEOUT` 由 `100`(1s) 改 `400`(4s)。
2. 重烧，重复阶段 2 的锁屏静置测试。
3. 对比 `0x08` 断连频率是否明显下降。
   - 下降明显 → 确认监督超时是主因，正式采用 4s（或 2~4s 折中，权衡功耗）。
   - 无变化 → 断连另有其因（信道/Doze/手机厂商策略），继续查 `ParamUpd` 是否被手机拒绝。

## 五、下一步（拿到数据后的修复方向 · 家庭多主人）

> 以下待阶段 1~3 数据确认后再动手，避免盲改。

1. **监督超时**：`DEFAULT_DESIRED_CONN_TIMEOUT` 1s→4s（阶段 3 验证后落地）。
2. **SNV 容量**（若走加密/系统配对）：`BLE_SNV_NUM` 扩容，容下多部手机 LTK，避免挤占。
3. **真·多 owner 应用层**（若要按手机独立授权/吊销）：
   - `Bonding_HandleBindCmd` 不再固定写 slot0，改为追加 / 按 key 去重；
   - `Bonding_HandleAuthResp` + `Bonding_VerifySignedCmd` 遍历全部 owner 的 bindKey 校验；
   - 对应 `docs` 待办「授权体系 v1」，届时 `fwsec` 升 2。
   - ⚠ 注意：若各手机共用同一绑定码，当前单 slot 已能多手机 AUTH；多 slot 仅在「每手机独立码/可单独吊销」时才必要。
4. **未鉴权 30s 强断**：多手机切换时若误伤，考虑延长或在 App 侧加快自动 AUTH。
