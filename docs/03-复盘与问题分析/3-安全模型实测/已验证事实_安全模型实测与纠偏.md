# 已验证事实：安全模型实测与纠偏（nrfconnect 复现 + 构建差异）

> 记录日期：2026-07-13
> 用途：把"用 nrfconnect 实测 + 真机观察"得到的**已验证事实**固化下来，避免后续开发（含 Codex 接手）重蹈对安全模型的错误假设。
> 结论均来自真机行为，非纯推理。涉及代码行号以 `文件:行` 标注。

---

## 0. 先厘清：两套独立的"门"

避免混淆，先把模型钉死：

| 通道 | 守护者 | 代码位置 | 现在是否强制 |
|---|---|---|---|
| **手动控制**（开锁/关锁/配置 FF03 指令） | `IsSessionAuthed`（需知道绑定码） | `peripheral.c:816` 硬 `DENY:AUTH_REQ` | ✅ 强制（固件侧） |
| **RSSI 近场自动解锁** | `LINK_ENCRYPTED \|\| IsSessionAuthed` | `keygo_core.c:442-443` | ⚠️ `LINK_ENCRYPTED` 分支**不查绑定码** |

关键点：**RSSI 自动解锁的 `LINK_ENCRYPTED` 分支，只认"链路加密了没"，不认"你有没有绑定码"**。链路加密来自"配对(bond)"，而配对当前是 Just Works（无密码）。这就是后面所有现象的开关。

## 1. 漏洞 A 实测复现：nrfconnect 配对 → RSSI 静默钥匙（无需绑定码）

### 复现步骤（用户实测）
1. 手机装 nrfconnect，连上 KeyGo 设备 → 因 `bonding.c:125-126` 是 `NO_INPUT_NO_OUTPUT + mitm=0`（Just Works），设备**无密码接受配对** → 手机与固件 SNV 各存一份 LTK（bond 建立）。
2. 打开 KeyGo App（或保持前台服务运行）→ App 连上设备，因已 bond 链路**自动加密** → `LINK_ENCRYPTED=true`。
3. 持手机走近车 → 固件 `keygo_core.c:442` 闸门 `LINK_ENCRYPTED` 满足 → **RSSI 自动解锁**，全程**没有输入过绑定码 '123456'**，手机甚至从未发过 `BIND:` 指令。

### 机制
nrfconnect 偷偷建了 bond → 之后任何进程维持的连接都加密 → 固件 RSSI 闸门把"加密连接"等同于"合法钥匙"，绕过 App 层 AUTH。

### ⚠ 严重性修正（重要，来自本轮新观察）
用户进一步实测：**App 关闭后，这台 nrfconnect 配对的手机并不会自动解锁**。

原因——RSSI 解锁的前提是**固件正在与一台"已连接"的手机测信号强度**。连接必须由某个进程维持：
- App 活着 → 前台服务 `Keygo-Foreground` 做后台扫描 + `tryAutoConnect` + 维持连接 → 有连接 → 有 RSSI → `LINK_ENCRYPTED` → 解锁。
- App 被杀 → 前台服务随之死亡（同进程）→ **无人维持连接** → 设备侧无连接、无 RSSI 样本 → 状态机到不了解锁 → 不解锁。

即：**裸 Android 系统对"已配对 BLE 设备"的后台自动重连 + 维持连接并不可靠/被省电策略限制**，实际重连靠的是 App 自己的前台服务，不是系统。

→ 漏洞 A 的实际利用门槛被重新定为：**攻击者需持有一台"正在运行 KeyGo App（或 nrfconnect）并保持连接"的手机，且此前用 nrfconnect 配过对**。路人、或 App 已关的手机，无法触发。严重性从"任何路人凑近"降为"装了 App 且蓄意配对的攻击者"。

## 2. "App 被杀也能走近解锁"——当前实测未生效（与 §1 同源）

方案 A 设计宣称（`Codex接手指南` §3 第 5 条）："`App 被杀+已配对+OS 重连加密仍能解锁"。

**实测结论：该收益当前不成立。** 根因同 §1——"OS 重连加密"在用户真机上不发生：
- 重连由被杀的前台服务驱动，App 一死，驱动没了；
- 裸系统不兜底维持已配对 BLE 连接。

所以 `keygo_core.c:442` 的 `LINK_ENCRYPTED` 闸门**逻辑在**，但**触发它的条件（系统级自动重连加密）在真机上不满足** → 表现为"车主手机杀 App 走近不解锁"。

> 对 Codex 的提醒：不要以为方案 A 已实现"无 App 解锁"。要实现它，得让前台服务成为**独立常驻进程**（`:remote`，不被 App 主进程杀死波及），或验证特定 ROM 的系统级 bond 自动重连——两者都是新工作，不是改一行的事。

## 3. HMAC 挑战应答被注释：BIND 明文传输

`app/BLE_Key_Go_App/stores/ble.js:2888-2890`：

```js
// ★ 阶段1(明文绑定): 控制类指令仅要求本连接已完成 BIND…
//   暂不走 NONCE/AUTH 加密握手。阶段3 恢复 ensureSession() 调用。
//   （原函数 ensureSession / _authWithKey 保留未删，阶段3 取消注释即可恢复加密路径。）
```

- App 端 HMAC `NONCE → AUTH:<hmac>` 握手被**故意关闭**；现在 `BIND:123456` 是**明文**直接发空中，`BIND:OK` 一到就把 `sessionAuthed=true`（`ble.js:3064`），没走加密握手。
- **注意**：固件侧 `crypto_sha256.c`（SHA256/HMAC 自研实现）**完好且自测通过**（`[CRYPTO] sha256 self-test: PASS`），现仅用于校验你发的 `BIND:123456` 对不对（KDF 比对）。被注释的只是 App 端"每次连接用 HMAC 重新证明持有密钥"那层握手，不是算法本身。
- 恢复加密握手成本低：取消注释 `ensureSession` / `_authWithKey`（阶段3）即可。但 `BIND:<码>` 那一次仍是明文——要彻底防嗅探，需配合"先配对再 BIND"（链路加密覆盖）或认证配对。

## 4. 构建路径差异：Android Studio 离线包 vs HBuilderX 自定义调试基座

用户实测新现象：**用 Android Studio（离线打包）发布后的 App，nrfconnect 配对后不会"自动验证绑定通过"；而 HBuilderX 自定义调试基座会。**

### 现象
- HBuilderX 自定义调试基座：nrfconnect 配对后，App 连接 → 链路加密 → 表现为"绑定验证通过"（§1 机制）。
- Android Studio 离线发布包：同样 nrfconnect 配对后，发布的 App **不**自动验证通过。

### 可能原因（⚠ 待验证，非定论）
项目有两条打包路线（`MEMORY.md` 铁律）：
- **(A) HBuilderX 云端自定义调试基座**：带 debug-server，aar 走 `nativeplugins/Keygo-Foreground/android/keygo-foreground.aar`。
- **(B) Android Studio 离线打包**：裸 APK（无 debug-server），**已知不一定吃到 `android/` 下最新 aar**，且 `manifest.json` 的 `versionCode` / `packageName` 可能与云端基座不同。

推测方向（需抓日志确认，不要当结论）：
1. **packageName / 签名不同** → 两条路的 App 在系统里是"两个应用"，BLE 绑定/连接状态（尤其 `createBond` 回调、`LinkEncrypted` 判定）相互隔离。
2. **离线包没吃到最新 `keygo-foreground.aar`**（已知坑）→ 旧 aar 的前台服务/配对回调行为不同，导致连接未加密或 `LINK_ENCRYPTED` 未被识别。
3. **离线发布包的前台服务未正确常驻/未触发重连** → 无活跃连接 → 无 `LINK_ENCRYPTED` → 不"验证通过"。

### 验证方法（给 Codex / 后续自查）
- 两套包都装，分别连设备，抓 `KeygoBleScanSvc|KeygoFgModule` 日志，对比：是否连上、是否 `LINK_ENCRYPTED`、BondState 回调。
- 确认离线包 `android/` 下 aar 时间戳与 `_build/source/*.java` 一致（不一致 = 没吃到新 aar，需重编 + versionCode+1）。

## 5. 综合结论与安全建议（不写代码，仅方向）

当前真实安全姿态：
- 手动控制：固件强制 AUTH（知道绑定码）→ 安全。
- RSSI 自动解锁：`LINK_ENCRYPTED` 可被 Just Works 配对绕过 → 漏洞 A（已被 nrfconnect 实测复现），但利用需"App 运行 + 曾配对"。
- 传输：`BIND/SETCODE` 明文（HMAC 握手被注释）→ 漏洞 B（可被嗅探，需抓包时机）。
- "App 被杀解锁"：未生效（重连依赖被杀的前台服务）。

最小成本加固（不依赖 CH582 passkey 能力）：
1. **调序：先配对，再发 `BIND`** → `BIND` 在加密链路上传 → 堵明文嗅探（漏洞 B）。改动在 `ble.js` `_triggerBond` 与 BIND 流程顺序。
2. **恢复 HMAC AUTH（取消注释阶段3）** → 即使链路被嗅，NONCE 一次性防重放、拿不到绑定码。
3. **封死漏洞 A（认证配对 / 弹框）** → 配对码 = 绑定码，不知道码配不上对。唯一命门：须先确证 CH582 支持"无头设备 + 固定 passcode + `mitm=1` 不回退 Just Works"（否则走 NFC/二维码 OOB）。这是设计文档 P0 的硬前提，落地前必须验证。
