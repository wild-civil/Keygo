# KeyGo v3.16 — CH582M 看门狗方案分析

> **一句话总结**：CH582M 硬件看门狗最长只能跑 557ms（半秒），太短了，可能在 BLE 正常操作时误复位。我们巧用 WWDG 的中断模式：**禁用硬件复位、启用中断，每次溢出（约 500ms）进入中断检查主循环是否活着，连续 5 次没报到（= 2.5s）才软件复位**。用看门狗本身的计时能力 + 软件计数器，既保留了硬件时钟源的可靠性，又达到了 2.5s 的安全超时。

## 1. 什么是看门狗？——小学生也能懂的故事

想象你养了一只狗，让它在院子里看门。

每天早上你来上班时，**给狗倒一碗水**（这叫"喂狗"）。

如果有一天你没来——狗没水喝了，过一段时间它就会**大声叫**，通知所有人"出事了！"

**把这个故事翻译到芯片上：**

| 故事里的概念 | 芯片上的对应 |
|-------------|-------------|
| 你每天来上班 | 芯片主循环每跑完一圈 |
| 给狗倒水 | 重置看门狗计数器（喂狗） |
| 狗没水喝的时间 | 看门狗超时时间 |
| 狗大声叫 | 芯片自动复位重启 |

**看门狗就是一个"主循环活着吗？"的哨兵**。主循环正常工作 → 哨兵一直沉默。主循环卡死了 → 哨兵过一段时间后触发重启。

## 2. 为什么需要看门狗？

### 2.1 车载设备的特殊要求

KeyGo 是一个**车载 BLE 钥匙**，塞在车里，没人看着它——不像手机 App 可以手动重启。

**可能的死机原因：**

| 死机原因 | 通俗解释 | 严重程度 |
|---------|---------|---------|
| 内存踩踏 | 某段代码写了不该写的内存地址，把程序搞乱了 | 高 |
| 死循环 | 某个 while 循环条件永远为真，出不来了 | 中 |
| BLE 协议栈卡死 | 射频中断丢失，连接处理函数永远在等一个不会来的事件 | 中 |
| 栈溢出 | 嵌套调用太深，栈空间用完了 | 低（我们已经够用） |

**没有看门狗 → 设备死了就死了，直到车主发现车钥匙失灵。**

**有看门狗 → 设备死了 2.5 秒后自动复活，车主根本感觉不到。**

## 3. CH582M 硬件看门狗的局限

### 3.1 CH582M 只有一个硬件看门狗：WWDG

CH582M 芯片里有一个叫 **WWDG（窗口看门狗）** 的硬件模块。

```
时钟源：系统主频 60MHz ÷ 131072 ≈ 457.8 Hz
每一拍  ≈ 2.18 毫秒
计数器  ：8 位（0~255）
最长超时：255 × 2.18ms ≈ 557ms
```

### 3.2 557ms 为什么太短？

问题出在 **BLE 连接建立**的环节。

当手机连接 KeyGo 设备时，会发生一系列操作：

```
手机发起连接
  → CH582M 收到 GAPROLE_CONNECTED 事件
    → 读取 Flash 里的绑定信息（SNV 数据）
    → 初始化连接参数
    → 配置加密
    → 通知 App 连接成功
```

其中 **Flash 读取**在某些边界条件下可能需要 **几百毫秒**（DataFlash 忙等待）。

**正常情况**：这个流程可能耗时 100~800ms

如果看门狗设定为 557ms，那么：
- 主循环在处理 BLE 连接 → 没空喂狗
- 557ms 到了 → 看门狗触发硬件复位
- **设备在正常的 BLE 连接过程中被误杀了！**

### 3.3 能不能在主循环中间喂狗？

不行。因为 BLE 连接处理是在 `TMOS_SystemProcess()` 函数**内部**完成的：

```
while(1) {
    TMOS_SystemProcess();   ← 这个函数内部可能需要 800ms
    喂狗();                  ← 到这时候已经来不及了
}
```

我们没法在 SDK 的 `TMOS_SystemProcess()` 内部插入喂狗代码——那是别人写的库函数。

## 4. 我们需要什么样的看门狗？

根据之前的分析，2.5 秒是最合理的超时时间：

| 超时时间 | 分析 |
|---------|------|
| **< 1s** | ❌ 太短 —— BLE 连接事件可能接近 1s，会误复位 |
| **2.5s** | ✅ 刚好 —— 覆盖所有正常操作 + 有安全余量 + 用户感知不到 |
| **> 10s** | ❌ 太长 —— 车主走到车边发现不工作，等 10s 太久了 |

**但 CH582M 的硬件看门狗最长只有 557ms，达不到 2.5s。**

## 5. 方案演进：三次迭代

### 5.1 第一版：硬件复位模式（557ms）

```
主循环每次喂狗 WWDG_SetCounter(255) → 557ms 内不喂狗 → 硬件自动复位
```

| 优点 | 缺点 |
|------|------|
| 硬件级别，中断关闭也能复位 | **557ms 太短，BLE 连接可能误触发** |
| 实现简单，只需两行代码 | 无法延长超时 |
| | 一旦误杀，设备不断重启，更糟 |

### 5.2 第二版：TMR0 定时器方案（被替换）

```
TMR0 定时器每 500ms 中断 → 检查主循环 alive 标志 → 连续 5 次失败 = 2.5s 复位
```

| 优点 | 缺点 |
|------|------|
| 2.5s 安全超时，不会误杀 | 需要额外占用 TMR0 外设 |
| 灵活可调 | TMR0 时钟源 = 系统主频（PLL 停振就瞎了） |
| | 假如后面要用 TMR0 做别的事，就冲突了 |

### 5.3 第三版（最终）：WWDG 中断 + 软件计数器

**核心洞察（用户提出）**：WWDG 有两杆"枪"，我们之前只用了其中一杆。

```
                    ┌── RB_WDOG_RST_EN ──→ 溢出后直接硬件复位（我们禁用这一杆）
WWDG 溢出 ────────┤
                    └── RB_WDOG_INT_EN ──→ 溢出后触发中断 ISR（我们启用这一杆！）
```

| 控制位 | 作用 | 本方案 |
|--------|------|--------|
| `RB_WDOG_RST_EN` | 溢出 → 芯片立即硬件复位 | **关闭**（太猛，557ms 误杀） |
| `RB_WDOG_INT_EN` | 溢出 → 进入 `WDOG_BAT_IRQHandler` 中断 | **开启**（先检查再决定） |

**改良思路**：
1. 让 WWDG 每 ~500ms 溢出一次（计数器填 26，要跑 229 拍才溢出）
2. 溢出后不进硬件复位，而是进入中断 ISR
3. ISR 里检查主循环的 alive 标志
4. 活着 → 清零计数器；死了 → 计数器 +1
5. 连续 5 次死 → 执行软件复位（2.5s）

## 6. 最终方案：WWDG 中断 + 软件计数器

### 6.1 为什么这是最优解？

| 考虑因素 | 第一版 (HW Reset) | 第二版 (TMR0) | 第三版 (WWDG-INT) ✅ |
|---------|------------------|---------------|---------------------|
| 超时时间 | 557ms（固定） | 2.5s（可调） | **2.5s（可调）** |
| BLE 连接误杀风险 | **高** ⚠️ | 无 ✅ | **无** ✅ |
| 硬件开销 | 0 个额外外设 | 占用 TMR0 | **0 个额外外设** |
| 时钟源 | Fsys/131072 | 系统主频 | **Fsys/131072**（和 WWDG 同一时钟） |
| 中断关闭时能救活 | ✅ | ❌ | ❌ |
| 概念清晰度 | 简单 | "用定时器模拟看门狗" | **"看门狗自己负责计数"** 🔥 |

**核心优势**：
- **复用硬件看门狗（WWDG）**：不需要额外占用 TMR0 定时器
- **用看门狗做看门狗的事**：逻辑自洽，一看就懂
- **只需重写已有的弱定义 ISR**：启动文件 `startup_CH583.S` 里 `WDOG_BAT_IRQHandler` 是 `weak` 定义（默认死循环），我们直接重写即可

### 6.2 关键参数计算

```
WWDG 时钟频率 = 系统主频 / 131072
             = 60MHz / 131072
             ≈ 457.8 Hz

每拍时间 = 1 / 457.8Hz ≈ 2.1845ms

计数器从 26 数到 255（8 位溢出）:
  拍数 = 255 - 26 = 229 拍
  时间 = 229 × 2.1845ms ≈ 500ms

连续 5 次不喂狗:
  总超时 = 5 × 500ms = 2.5s  ✅
```

### 6.3 工作流程（一步步）

```
[初始化]
  WWDG_ResetCfg(DISABLE)     ← 拔掉"盲目复位"的线
  WWDG_ITCfg(ENABLE)         ← 接上"先检查再决定"的线
  WWDG_SetCounter(26)        ← 加载计数器初值 → ~500ms 后溢出

[主循环 — 每次跑完一圈]
  g_mainLoopAlive = 1;       ← "我来过了！"（不再喂狗，只举手报到）

[WWDG 中断 — 每 ~500ms 自动触发]
  WDOG_BAT_IRQHandler:
    if (主循环报到了?) {
        g_mainLoopAlive = 0;     ← 清标志，等下一轮
        g_watchdogCount = 0;     ← 重置死亡计数器
    } else {
        g_watchdogCount++;       ← 又一周期没报到
        if (g_watchdogCount >= 5) {
            GPIO 安全清理        ← 拉低所有控制引脚
            SYS_ResetExecute();  ← 软件复位！
        }
    }
    WWDG_SetCounter(26);     ← 重新装填，下一个 ~500ms 周期
```

### 7.2 ISR 完整代码（带逐行注释）

```c
/*********************************************************************
 * @fn      WDOG_BAT_IRQHandler
 *
 * @brief   WWDG 看门狗溢出中断服务函数
 *
 * @说明
 *   【为什么这个函数会每 500ms 被调用？】
 *   WWDG 计数器从 26 往上数，每 2.18ms 加 1，229 拍后溢出（~500ms）。
 *   溢出后会触发这个中断函数（因为我们开启了 RB_WDOG_INT_EN）。
 *
 *   【调用频率】
 *   ~2 Hz（每 500ms 一次）
 *
 *   【执行时间】
 *   < 10μs（纯寄存器操作 + 最多一个软件复位）
 *
 *   【注意】
 *   - 启动文件的 WDOG_BAT_IRQHandler 是 __weak 定义（死循环），
 *     我们在此重写它。
 *   - ISR 内必须清除中断标志 WWDG_ClearFlag()，否则会反复触发。
 *   - WWDG_SetCounter(26) 重新装填计数器，启动下一个 ~500ms 倒计时。
 *********************************************************************/
__INTERRUPT
__HIGH_CODE
void WDOG_BAT_IRQHandler(void)
{
    /* ── 步骤 1: 确认是 WWDG 溢出触发的（不是其他共用中断源）── */
    if (WWDG_GetFlowFlag())
    {
        /* ── 步骤 2: 清除溢出标志（必须！否则退出 ISR 后立即再次触发）── */
        WWDG_ClearFlag();

        /* ── 步骤 3: 判断主循环是否还活着 ── */
        if (g_mainLoopAlive != 0)
        {
            /* 主循环在此周期内设置过 alive 标志 → 一切正常 */
            g_mainLoopAlive = 0;    /* 清除标志，等待下一轮 */
            g_watchdogCount = 0;    /* 死亡计数器归零 */
        }
        else
        {
            /* 主循环没有报到 → 这一周期可能卡死了 */
            g_watchdogCount++;

            if (g_watchdogCount >= 5)
            {
                /* 连续 5 个周期（~2.5s）主循环没反应 → 确认死机
                 *
                 * ── 复位前安全措施 ──
                 * 把所有 GPIO 控制引脚拉低，防止：
                 *   - 复位过程中引脚电平抖动导致车锁误动作
                 *   - 复位后 I/O 默认态不确定导致短暂异常输出
                 */
                GPIOA_ResetBits(GPIO_Pin_4 | GPIO_Pin_5 | GPIO_Pin_6 | GPIO_Pin_7);
                GPIOB_ResetBits(GPIO_Pin_4);

                /* 执行软件复位 — 等价于芯片完全重启 */
                SYS_ResetExecute();
            }
        }

        /* ── 步骤 4: 重新装填计数器，启动下一个 ~500ms 周期 ──
         *    26 → 溢出时间 = (255-26) × 2.18ms ≈ 500ms          */
        WWDG_SetCounter(26);
    }
}
```

### 7.3 主循环改动

```c
// ──────────── 文件顶部 — 全局变量 ────────────
// 【软看门狗】主循环存活标志 & 死亡计数器
// - g_mainLoopAlive: 主循环每跑完一圈设为 1，ISR 读到后清零
// - g_watchdogCount: ISR 连续未读到 alive 的次数，达到 5 即复位
// - 两个变量都被 ISR 和主循环同时访问，必须用 volatile 防止编译器优化
volatile uint8_t g_mainLoopAlive = 0;
volatile uint8_t g_watchdogCount = 0;

// ──────────── 主循环 ────────────
void Main_Circulation(void)
{
    while(1)
    {
        TMOS_SystemProcess();        // BLE 协议栈 & 应用事件处理
        g_mainLoopAlive = 1;         // ← "我还活着！" — 替代原来的喂狗
    }
}
```

### 7.4 初始化改动（main 函数中）

```c
int main(void)
{
    SetSysClock(CLK_SOURCE_PLL_60MHz);

    /* ★ v3.16: WWDG 中断模式 — 软看门狗 2.5s
     * ────────────────────────────────────────────────────────
     * 不复位！只中断！500ms 溢出 → ISR 里检查 → 连续 5 次才复位
     * 详见 docs/KeyGo_v3.16_看门狗方案分析.md 第 6 章
     * ──────────────────────────────────────────────────────── */
    WWDG_ResetCfg(DISABLE);      // 溢出后不硬件复位（太猛，557ms 会误杀正常 BLE 操作）
    WWDG_ITCfg(ENABLE);          // 溢出后触发 WDOG_BAT_IRQHandler 中断
    WWDG_SetCounter(26);         // 初值 26 → ~500ms 后第一次溢出

    // ... 其余初始化不变 ...
    CH58X_BLEInit();
    HAL_Init();
    GAPRole_PeripheralInit();
    Peripheral_Init();
    Main_Circulation();
}
```

## 9. 风险与边界条件

### 9.1 什么情况下会生效？

| 故障场景 | 是否生效 | 详细 |
|---------|---------|------|
| 主循环某个 while/for 里无限循环 | ✅ | 主循环没设置 alive → ISR 正常触发 → 5 次后复位 |
| BLE 协议栈内部死锁（TMOS_SystemProcess 不返回） | ✅ | 同理 |
| 内存踩踏导致 PC 跑飞（跳到未知地址） | ✅ | 不会回到主循环设置 alive |
| **全局中断关闭 + 上述任一故障** | ❌ | WWDG 的 ISR 也进不去 |
| **芯片 60MHz 晶振停振** | ❌ | WWDG 时钟源是 Fsys/131072，晶振没了时钟也没了 |

### 9.2 为什么"中断关闭"不太可能？

CH582M 的 BLE 协议栈严重依赖定时器中断处理射频时序。如果全局中断被长时间关闭：
1. BLE 连接会首先断开（射频中断丢失）
2. 用户已经发现问题，不需要等看门狗
3. 这个场景几乎只在代码 Bug 中出现（如 `__disable_irq()` 后忘记恢复）

### 9.3 为什么"晶振停振"也不需要考虑？

晶振停振 = 芯片完全停止工作 = 程序根本不执行 = 看门狗有没有都一样。这种情况下 BLE 早就断了，用户会察觉。

### 9.4 2.5s 对各种操作的容错分析

| 操作 | 典型耗时 | 对 2.5s 的影响 |
|------|---------|---------------|
| 主循环空转 | < 1ms | 几乎为零 |
| BLE 广播 | < 10ms | 几乎为零 |
| BLE 连接事件（含 DataFlash 读） | 100~800ms | 安全，远低于 2.5s |
| RSSI 采样 + 卡尔曼滤波 | < 100ms | 几乎为零 |
| GPIO 操作 | < 1ms | 几乎为零 |
| **Flash 擦除/写入** | **可能 1~2s** | ⚠️ 接近边界 |
| **启动初始化（BLEInit + HAL_Init + Peripheral_Init）** | **可能 ~500ms** | ⚠️ 开机时可能溢出 1~2 次 |

> **关于 Flash 操作**：当前 KeyGo 固件中没有主动的 Flash 擦除/写入操作。但如果 SBP 安全配置写入（v3.15）涉及 Flash 操作且超过 2.5s，将 `g_watchdogCount >= 5` 改为 `>= 8`（= 4s）即可。

> **关于启动初始化**：开机时主循环还没开始跑，g_mainLoopAlive 初始化 = 0。启动初始化如果超过 500ms，WWDG 可能溢出 1~2 次。由于我们在 main() 中 **先初始化 WWDG，再进行 BLE/HAL 初始化**，启动过程中 g_mainLoopAlive = 0 → g_watchdogCount 会上涨。但 BLEInit+HAL_Init+Peripheral_Init 通常 < 500ms，所以问题不大。如果实测发现启动时误复位，在时基初始化前加一行 `g_mainLoopAlive = 1` 即可。

## 10. 总结

```
提问：为什么不用 CH582M 自带硬件看门狗的复位功能？
回答：WWDG 的硬件复位模式最长只能撑 557ms，
     但 BLE 连接事件有时需要 800ms（含 DataFlash 读），
     557ms 会在正常操作时把设备误杀掉。

提问：那 2.5s 的看门狗怎么实现？
回答：巧用 WWDG 的两根控制线——
     禁用硬件复位那根（RB_WDOG_RST_EN），只保留中断那根（RB_WDOG_INT_EN）。
     WWDG 每 ~500ms 溢出一次 → 进入 ISR → 检查主循环 alive 标志。
     连续 5 次 = 2.5s 没反应 → 才软件复位。
     本质上是"同一个看门狗硬件" + "软件计数器"的优雅组合。

提问：为什么不用 TMR0 定时器来做这个？
回答：WWDG 本身就有时钟源和溢出中断能力，何必多占一个 TMR0？
     而且 WWDG 用的是 Fsys/131072 时钟，和硬件复位模式完全相同的时钟路径，
     比 TMR0 的"用系统主频模拟看门狗"逻辑更自洽。
     多余出来的 TMR0 可以留给未来的其他功能。

提问：这个方案最妙的地方在哪？
回答：WWDG 本身的设计就是为了当看门狗，
     我们只是换了一种"叫醒方式"——
     从"溢出后一刀切掉"变成"溢出后先看一眼主循环还活着没"。
     从而即利用了 WWDG 精准的硬件时钟，
     又突破了 8 位计数器只能到 557ms 的物理限制。
```
